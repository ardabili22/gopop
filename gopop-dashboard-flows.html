🔐 Login Admin › F01
Login & Masuk Dashboard
Titik masuk pertama semua anggota tim internal — dari Super Admin sampai CS. Login menggunakan email dan password, bukan nomor HP.
👥 Semua role admin 🔔 Buka gopop-lilac.vercel.app ⏱ <30 detik
1
Alur Normal
▾
1
Admin buka dashboard di browser
Ketik alamat dashboard di browser. Kalau belum pernah login atau sesi sudah habis, langsung diarahkan ke halaman login.
2
Masukkan email dan password
Admin isi email kerja dan password. Email bersifat unik per anggota tim — tidak bisa dua orang pakai email yang sama.
3
Klik "Masuk" — sistem verifikasi identitas
Sistem cek apakah email terdaftar dan password cocok. Kalau benar, sistem buat sesi login untuk admin ini.
4
Dashboard terbuka sesuai role
Admin masuk ke halaman utama dashboard. Menu yang muncul di sidebar berbeda-beda tergantung role:

  • Super Admin: Lihat dan akses semua menu
  • Ops (Acoy): Monitoring, Komplain, Reseller, Rekonsiliasi, Produk
  • Finance (Agung): Monitoring (read), Rekonsiliasi, Laporan
  • CS: Komplain, Pengguna (read-only), Monitoring (read-only)
  • IT: Semua menu untuk keperluan teknikal
2
Aturan Bisnis
▾
📌
Login admin menggunakan email, bukan nomor HP. Ini berbeda dari login consumer/reseller yang pakai nomor HP.
📌
Setelah 5 kali salah password dalam satu sesi, akun terkunci sementara selama 30 menit. Super Admin bisa unlock lebih cepat dari menu Role & Akses.
📌
Sesi login aktif selama 8 jam. Setelah itu, admin harus login ulang. Tidak ada opsi "ingat saya" untuk alasan keamanan.
📌
Satu akun admin tidak boleh dipakai bergantian oleh dua orang berbeda. Setiap anggota tim punya akun masing-masing agar semua aksi tercatat per individu.
3
Bad Cases
▾
Password salah berkali-kali — akun terkunci
"Akun kamu terkunci sementara. Coba lagi dalam 30 menit atau hubungi Super Admin untuk buka lebih cepat."
Super Admin buka dari menu Role & Akses → cari nama admin → klik "Unlock Akun". Atau tunggu 30 menit otomatis terbuka.
PIC: Super Admin (Acoy) untuk unlock manual
Admin lupa password
Klik "Lupa Password" di halaman login → isi email → cek email untuk link reset.
Link reset password dikirim ke email terdaftar, berlaku 1 jam. Kalau email tidak masuk, cek folder spam atau minta Super Admin reset manual dari dashboard.
PIC: Admin sendiri (via email) atau Super Admin (reset manual)
Admin baru tidak bisa login — akun belum dibuat
"Email tidak terdaftar."
Super Admin buat akun baru dari menu Role & Akses → Tim → Tambah Anggota. Isi email, tentukan role, dan kirim undangan. Admin baru dapat email untuk set password pertama kali.
PIC: Super Admin (Acoy)
4
Bottleneck
▾
Satu akun dipakai bergantian oleh beberapa orang — tidak ada jejak siapa yang melakukan apa
Tegaskan aturan: satu akun = satu orang. Kalau ada anggota baru, buatkan akun baru jangan pinjam akun lama. Ini penting karena semua aksi di dashboard (refund, suspend reseller, update status transaksi) dicatat atas nama akun yang login.
PIC: Acoy (pastikan SOP ini dijalankan)
Admin tidak logout saat meninggalkan komputer — risiko akses tidak sah
Sistem otomatis logout setelah tidak ada aktivitas selama 2 jam. Tapi lebih baik admin selalu logout manual saat meninggalkan meja. Tambahkan kebiasaan ini ke SOP onboarding tim.
PIC: Acoy (SOP tim)
5
Hasil Akhir
▾
✅ Flow Sukses
Admin masuk ke dashboard dengan menu sesuai role-nya
Semua aksi admin tercatat atas nama akun yang login
Sesi aktif 8 jam, setelah itu perlu login ulang
❌ Flow Gagal
Akun terkunci karena salah password 5x — perlu tunggu 30 menit atau minta Super Admin unlock
Email tidak terdaftar — perlu Super Admin buat akun dulu

Sesi Habis / Auto-Logout
Sesi login admin tidak berlangsung selamanya. Flow ini menjelaskan apa yang terjadi saat sesi habis dan bagaimana admin kembali masuk tanpa kehilangan pekerjaannya.
👥 Semua role admin 🔔 Sesi 8 jam habis atau idle 2 jam
1
Alur Normal
▾
1
Sistem deteksi sesi admin akan habis
15 menit sebelum sesi berakhir, muncul notifikasi kecil di pojok kanan atas: "Sesi kamu akan berakhir dalam 15 menit. Klik di sini untuk perpanjang."
2
Admin bisa perpanjang sesi
Klik notifikasi → sesi diperpanjang 8 jam lagi tanpa perlu input password ulang. Ini untuk menghindari admin terpaksa login ulang di tengah pekerjaan penting.
3
Kalau tidak diperpanjang — sesi habis
Dashboard otomatis redirect ke halaman login dengan pesan: "Sesi kamu sudah berakhir. Silakan masuk kembali untuk melanjutkan." Halaman yang sedang dibuka sebelumnya disimpan — setelah login, admin diarahkan kembali ke halaman yang sama.
4
Admin login ulang
Masukkan password → langsung kembali ke halaman yang sedang dikerjakan sebelum sesi habis.
2
Aturan Bisnis
▾
📌
Sesi aktif maksimal 8 jam sejak login terakhir, atau 2 jam sejak terakhir ada aktivitas (whichever comes first).
📌
Saat sesi habis di tengah pengisian form (misal sedang isi form refund), data yang belum disimpan akan hilang. Admin perlu isi ulang. Ini trade-off keamanan yang disengaja.
📌
Admin bisa logout manual kapan saja dari menu profil di pojok kanan atas. Disarankan selalu logout saat meninggalkan komputer.
3
Bad Cases
▾
Admin sedang proses refund besar — tiba-tiba sesi habis sebelum submit
Dashboard redirect ke login. Data refund yang sedang diisi hilang.
Admin login ulang → kembali ke halaman refund → isi ulang data. Ke depannya, sistem perlu simpan draft form secara otomatis setiap 30 detik untuk menghindari kehilangan data.
PIC: Isal + Ipong (implementasi auto-save draft)
4
Bottleneck
▾
Admin tidak tahu sesi akan habis — tiba-tiba ter-logout saat sedang kerja
Notifikasi 15 menit sebelum habis harus jelas dan tidak mudah terlewat. Kalau admin sedang fokus ke pekerjaan lain, notifikasi perlu ada suara atau visual yang mencolok bukan hanya teks kecil.
PIC: Isal + Ipong (perbaiki UI notifikasi sesi)
5
Hasil Akhir
▾
✅ Flow Sukses
Admin berhasil perpanjang sesi tanpa login ulang
Atau: login ulang dan kembali ke halaman yang sama
❌ Flow Gagal
Data form yang sedang diisi hilang karena sesi habis mendadak

Lihat & Filter Transaksi
Halaman utama Monitoring — tempat Rifky dan Acoy memantau semua transaksi yang masuk. Bisa disaring berdasarkan status, tanggal, produk, dan kata kunci.
👥 Rifky · Acoy · CS (read) 🔔 Setiap hari — pemantauan rutin
1
Alur Normal
▾
1
Buka menu Monitoring Transaksi
Klik menu "Monitoring" di sidebar. Halaman langsung menampilkan daftar transaksi terbaru, diurutkan dari yang paling baru di atas. Ada ringkasan angka di bagian atas: total transaksi hari ini, berapa yang sukses, gagal, dan sedang diproses.
2
Gunakan filter untuk mempersempit hasil
Tersedia filter:
  • Status: Semua / Menunggu Pembayaran / Diproses / Sukses / Gagal / Refunded / Dibatalkan / Perlu Review Manual
  • Tanggal: Hari ini / 7 hari / 30 hari / atau pilih rentang custom
  • Produk: Pulsa, Token Listrik, Transfer E-Wallet, dll
  • Pencarian: Ketik ID transaksi atau nomor HP user
3
Lihat daftar transaksi yang terfilter
Setiap baris menampilkan: waktu transaksi, nama user dan nomor HP, produk yang dibeli, nominal, status saat ini, dan nama supplier (biller) yang dipakai. Klik baris mana saja untuk lihat detail lengkapnya.
4
Export ke CSV kalau perlu
Klik tombol "Export CSV" di pojok kanan atas untuk download semua data yang sedang ditampilkan (sesuai filter aktif) ke dalam file Excel. Berguna untuk laporan harian Rifky.
2
Aturan Bisnis
▾
📌
CS hanya bisa lihat, tidak bisa aksi. CS tidak punya tombol refund, retry, atau update status. Mereka hanya bisa baca informasi untuk bantu jawab pertanyaan user.
📌
Status "Perlu Review Manual" artinya ada sesuatu yang tidak bisa diselesaikan otomatis oleh sistem — transaksi ini butuh perhatian Rifky atau Acoy segera.
📌
Data transaksi tersedia real-time. Tidak perlu refresh manual — halaman update otomatis setiap 30 detik selama halaman dibuka.
3
Bad Cases
▾
Filter tidak menghasilkan transaksi yang dicari — ID transaksi tidak ditemukan
Tabel kosong dengan pesan "Tidak ada transaksi yang cocok."
Pastikan ID transaksi yang diketik benar (bisa cek dari screenshot atau WA yang diterima user). Coba juga cari dengan nomor HP user sebagai alternatif. Kalau masih tidak ketemu, mungkin transaksi belum selesai dibuat (status draft).
PIC: CS atau Rifky yang sedang investigasi
Terlalu banyak transaksi "Perlu Review Manual" menumpuk
—
Ini tanda ada masalah di sisi supplier atau payment gateway. Rifky dan Acoy perlu segera cek satu per satu. Kalau jumlahnya banyak sekaligus, kemungkinan ada gangguan massal dari satu supplier — perlu koordinasi dengan Ipong untuk switch ke supplier backup.
PIC: Rifky + Acoy (investigasi) · Ipong (koordinasi supplier)
4
Bottleneck
▾
Tidak ada tab khusus untuk transaksi yang butuh tindakan segera — tercampur dengan semua transaksi
Perlu ditambahkan tab "Perlu Perhatian" yang otomatis menampilkan: transaksi berstatus "Perlu Review Manual", transaksi yang sudah "Diproses" lebih dari 15 menit tanpa update, dan transaksi "Gagal" yang belum di-refund. Rifky tidak perlu filter manual setiap kali — tinggal klik satu tab.
PIC: Isal + Ipong (tambah tab khusus di Monitoring)
5
Hasil Akhir
▾
✅ Flow Sukses
Tim punya gambaran jelas kondisi transaksi hari ini
Transaksi bermasalah teridentifikasi dan bisa segera ditangani
Export CSV tersedia untuk laporan
❌ Flow Gagal
Transaksi yang dicari tidak ditemukan karena ID salah atau data belum sync

📊 Monitoring Transaksi › F02
Lihat Detail Transaksi
Setelah menemukan transaksi yang dicari, admin buka detail lengkapnya. Di sini terlihat seluruh riwayat apa yang terjadi pada transaksi tersebut dari awal sampai akhir.
👥 Rifky · Acoy · CS 🔔 Klik baris transaksi di Monitoring
1
Alur Normal
▾
1
Klik baris transaksi — halaman detail terbuka
Halaman detail menampilkan semua informasi transaksi dalam satu tampilan.
2
Baca informasi transaksi
Informasi yang tersedia:
  • Identitas: ID transaksi, waktu dibuat
  • User: Nama, nomor HP, tipe akun (consumer/reseller/guest)
  • Produk: Nama produk, nominal, tujuan (nomor HP/ID pelanggan)
  • Pembayaran: Metode bayar, jumlah yang dibayar, waktu bayar
  • Supplier: Nama biller yang dipakai, referensi dari supplier
  • Status: Status saat ini beserta waktu terakhir berubah
3
Lihat timeline riwayat transaksi
Di bawah info utama ada timeline yang menampilkan setiap perubahan status dari awal sampai sekarang. Contoh: 14:30 Transaksi dibuat → 14:31 Pembayaran diterima → 14:31 Dikirim ke supplier → 14:32 Sukses.
Kalau ada admin yang pernah lakukan aksi manual (retry, refund, update status), juga tercatat di sini beserta nama adminnya.
2
Aturan Bisnis
▾
📌
Detail transaksi bersifat read-only untuk CS. Mereka bisa lihat semua info tapi tidak ada tombol aksi. Tombol aksi (refund, retry, update status) hanya muncul untuk Ops dan Super Admin.
📌
Timeline riwayat tidak bisa dimanipulasi. Semua yang tercatat di timeline adalah fakta permanen — berguna sebagai bukti kalau ada dispute dengan user atau supplier.
📌
Kalau transaksi punya tiket komplain terkait, ada link langsung ke tiket tersebut di halaman detail. CS tidak perlu cari manual di menu Komplain.
3
Bad Cases
▾
Timeline kosong atau tidak lengkap — tidak ada riwayat yang bisa dibaca
—
Ini tanda ada masalah teknikal dalam pencatatan riwayat transaksi. Rifky perlu catat ID transaksi yang bermasalah dan laporkan ke Irwan/Rifky BE untuk investigasi. Sementara itu, status transaksi masih bisa dibaca dari field status utama.
PIC: Irwan + Rifky (investigasi pencatatan timeline)
4
Bottleneck
▾
CS perlu tahu detail transaksi tapi tidak punya akses — harus minta ke Rifky setiap saat
CS seharusnya bisa lihat detail transaksi (read-only) tanpa perlu koordinasi. Akses read-only ke Monitoring sudah cukup — CS tidak perlu bisa aksi apapun. Ini mengurangi beban Rifky dari pertanyaan yang bisa dijawab sendiri oleh CS.
PIC: Acoy (pastikan role CS punya akses read-only ke Monitoring)
5
Hasil Akhir
▾
✅ Flow Sukses
Admin punya gambaran lengkap apa yang terjadi pada transaksi ini
Timeline lengkap tersedia sebagai bukti audit
Link ke tiket komplain terkait mudah diakses
❌ Flow Gagal
Timeline kosong — informasi riwayat tidak bisa dibaca

F03: Handle Transaksi Bermasalah →
📊 Monitoring Transaksi › F03
Handle Transaksi Bermasalah
Transaksi dengan status "Perlu Review Manual" atau yang stuck terlalu lama di "Diproses" perlu ditangani secara manual oleh Acoy atau Rifky. Flow ini menjelaskan langkah-langkahnya.
👥 Acoy · Rifky 🔔 Transaksi stuck / Perlu Review Manual ⏱ Target selesai dalam 1 jam kerja
1
Alur Normal
▾
1
Identifikasi transaksi yang bermasalah
Buka tab "Perlu Perhatian" di Monitoring — semua transaksi yang butuh tindakan sudah terkumpul di sini. Atau cari spesifik dengan filter status "Perlu Review Manual".
2
Buka detail dan pahami situasinya
Baca timeline transaksi. Lihat di mana tepatnya macet: apakah macet di pembayaran, di pengiriman ke supplier, atau supplier tidak memberi konfirmasi? Informasi ini menentukan aksi apa yang tepat.
3
Pilih aksi yang tepat
Ada beberapa pilihan aksi tergantung situasi:
  • Cek Status ke Supplier: Tanya langsung ke sistem supplier apakah transaksi ini sukses atau gagal di sisi mereka
  • Tandai Sukses Manual: Kalau sudah konfirmasi dari supplier bahwa produk terkirim tapi sistem belum update — gunakan ini. Wajib isi alasan.
  • Tandai Gagal & Proses Refund: Kalau sudah dipastikan gagal — ubah status ke gagal dan langsung proses refund ke user
4
Isi alasan aksi — submit
Setiap aksi manual wajib disertai alasan singkat (minimal 10 kata). Ini penting untuk audit trail. Contoh: "Konfirmasi dari Digiflazz via email bahwa pulsa sudah terkirim jam 14:35, referensi #DGF-88821."
5
Sistem catat aksi dan update status
Status transaksi berubah. Aksi Rifky/Acoy tercatat di timeline beserta waktu dan nama. Kalau ada tiket komplain terkait transaksi ini, status tiket juga otomatis diupdate.
2
Aturan Bisnis
▾
📌
Wajib konfirmasi ke supplier sebelum tandai sukses manual. Jangan pernah tandai transaksi sukses hanya berdasarkan asumsi — harus ada bukti dari supplier bahwa produk benar-benar terkirim.
📌
Aksi "Cek Status ke Supplier" bisa dilakukan maksimal 3 kali dalam 1 jam untuk transaksi yang sama. Lebih dari itu berarti ada masalah serius di supplier yang perlu Ipong tangani langsung.
📌
Kalau memilih "Tandai Gagal", refund akan diproses otomatis jika user sudah membayar. Tidak perlu proses refund terpisah.
📌
Semua aksi manual dicatat permanen. Tidak bisa dihapus atau diubah setelah disubmit. Pastikan yakin sebelum klik submit.
3
Bad Cases
▾
Rifky tandai sukses manual — ternyata produk tidak terkirim ke user
User WA ke CS: "Pulsa saya belum masuk tapi status sudah Sukses."
Ini situasi kritis. CS eskalasi ke Acoy segera. Acoy perlu hubungi supplier langsung untuk klarifikasi. Kalau terbukti produk tidak terkirim, perlu koordinasi dengan Agung untuk proses kompensasi manual ke user. Timeline di sistem akan tercatat semua perubahan ini sebagai bukti.
PIC: Acoy (koordinasi) · Agung (kompensasi) · Ipong (klarifikasi ke supplier)
Transaksi sudah ditandai selesai tapi refund tidak bisa diproses karena nominal tidak cocok
—
Ini terjadi kalau ada selisih antara nominal yang dibayar user dan nominal di sistem. Agung perlu investigasi manual dan proses refund via metode alternatif sambil koordinasi dengan Irwan untuk investigasi penyebab selisih.
PIC: Agung + Irwan + Rifky
4
Bottleneck
▾
Tidak ada standar berapa lama transaksi boleh stuck sebelum perlu diintervensi manual
Perlu ada SLA yang jelas: transaksi "Diproses" lebih dari 15 menit otomatis masuk ke tab "Perlu Perhatian" dan Rifky mendapat notifikasi. Tanpa standar ini, transaksi bisa stuck berhari-hari tanpa ada yang handle.
PIC: Acoy (tetapkan SLA) · Irwan + Rifky (implementasi notifikasi otomatis)
Rifky tidak tahu harus hubungi siapa di supplier untuk klarifikasi cepat
Buat dan simpan kontak PIC dari setiap supplier (Digiflazz) di tempat yang mudah diakses tim — bisa di halaman Produk & Harga atau dokumen internal. Jangan sampai waktu terbuang hanya untuk cari kontak saat situasi darurat.
PIC: Ipong (maintain kontak supplier)
5
Hasil Akhir
▾
✅ Flow Sukses
Transaksi berhasil diklarifikasi dan status diupdate dengan tepat
Aksi tercatat lengkap di timeline sebagai audit trail
User mendapat kepastian — sukses atau refund diproses
❌ Flow Gagal
Tandai sukses tapi produk tidak terkirim — user komplain
Supplier tidak responsif — status tidak bisa dikonfirmasi

📊 Monitoring Transaksi › F04
Proses Refund Manual
Refund manual dilakukan ketika refund otomatis gagal atau tidak berjalan. Ini adalah proses yang melibatkan Agung (Finance) karena menyangkut pengembalian dana ke user.
👥 Agung (Finance) · Acoy (approve) 🔔 Refund otomatis gagal atau transaksi ditandai Gagal ⏱ Target: Dana kembali dalam 1×24 jam
1
Alur Normal
▾
1
Identifikasi transaksi yang perlu refund
Bisa berasal dari dua jalur: (1) CS eskalasi tiket komplain yang butuh refund, atau (2) Rifky/Acoy menemukan transaksi gagal yang belum di-refund saat monitoring rutin.
2
Buka detail transaksi — klik "Proses Refund"
Tombol "Proses Refund" hanya muncul kalau: (1) transaksi statusnya Gagal, dan (2) user sudah membayar (status pembayaran bukan Expired atau Dibatalkan). Kalau tidak memenuhi syarat, tombol tidak muncul.
3
Isi detail refund dan submit
Form refund berisi:
  • Nominal refund: Terisi otomatis sesuai jumlah yang dibayar user. Agung bisa ubah kalau ada alasan khusus (refund parsial).
  • Alasan refund: Wajib diisi. Contoh: "Transaksi gagal dari sisi supplier Digiflazz, produk tidak terkirim."
4
Sistem proses refund via payment gateway
Sistem kirim permintaan refund ke Midtrans/DOKU sesuai metode bayar asal. Status transaksi berubah ke "Refunded". WA otomatis dikirim ke user: "Dana Rp [nominal] sedang kami kembalikan ke [metode bayar]. Estimasi masuk 1–3 hari kerja."
5
Monitor konfirmasi refund selesai
Beberapa jam kemudian, payment gateway mengirim konfirmasi bahwa refund sudah selesai. Status transaksi berubah final. WA kedua dikirim ke user: "Dana sudah kembali ke akun kamu."
2
Aturan Bisnis
▾
📌
GoPop tidak memotong biaya apapun saat refund. User mendapat kembali 100% jumlah yang dibayar. Biaya admin yang sudah dibayar ke payment gateway tidak ditagihkan ke user.
📌
Refund hanya bisa diproses ke metode bayar yang sama dengan yang dipakai saat transaksi. Tidak bisa refund ke rekening berbeda atau metode berbeda.
📌
Refund untuk transaksi yang sudah berstatus Sukses tidak bisa diproses dari dashboard biasa — butuh persetujuan Super Admin dan prosedur khusus karena berarti ada dispute tentang kualitas produk.
📌
Satu transaksi hanya bisa di-refund satu kali. Sistem akan tolak percobaan refund kedua untuk transaksi yang sama.
3
Bad Cases
▾
Refund gagal di sisi payment gateway — Midtrans/DOKU menolak permintaan
User belum menerima dana kembali meski sudah dijanjikan.
Agung cek alasan penolakan dari log dashboard. Biasanya karena akun user sudah tidak aktif atau metode bayar sudah kadaluarsa. Agung perlu hubungi user via WA untuk cari solusi alternatif (transfer manual ke rekening user). Koordinasi dengan Irwan untuk investigasi kenapa gateway menolak.
PIC: Agung (handle ke user) · Irwan (investigasi gateway)
User claim refund tidak masuk setelah 3 hari kerja
User WA CS: "Katanya sudah direfund tapi tidak masuk ke OVO saya."

  • Match: Ada di kedua sumber, nominal sama — ini yang seharusnya
  • Hanya di GoPop: Tercatat SUCCESS di GoPop tapi tidak ada di laporan gateway — mungkin gateway belum settle atau ada transaksi yang lolos
  • Hanya di Gateway: Ada di laporan gateway tapi tidak ada di sistem GoPop — ini yang paling berbahaya, berarti ada uang masuk yang tidak tercatat
3
Rifky review hasil dan tandai status
Kalau semua match → klik "Approve Rekonsiliasi". Kalau ada yang tidak match → masuk ke F03 Selisih untuk investigasi lebih lanjut. Rekonsiliasi yang sudah di-approve tercatat permanen dan tidak bisa diubah.
4
Agung approve sebagai Finance
Rifky selesai rekonsiliasi → notifikasi ke Agung untuk approval final. Agung review summary → approve → rekonsiliasi periode ini resmi selesai.

▾

📌

Investigasi & Penyelesaian Selisih Settlement
Ketika rekonsiliasi menemukan angka yang tidak cocok antara sistem GoPop dan laporan gateway. Ini kondisi yang jarang tapi serius — setiap selisih harus diselesaikan dan tidak boleh diabaikan.
👥 Rifky (investigasi) · Agung (Finance — keputusan) · Irwan (debug teknikal) 🔔 Ditemukan selisih saat rekonsiliasi F02 ⏱ Target resolusi: 1×24 jam kerja
1
Tiga Jenis Selisih & Cara Handle
▾
🔴
Jenis 1 — Ada di Gateway, tidak ada di GoPop (paling serius):
Berarti ada uang masuk dari user yang tidak terproseskan. Kemungkinan penyebab: webhook dari gateway gagal diterima sistem GoPop, transaksi duplicate di sisi gateway, atau bug di backend.

Cara handle: Irwan cek webhook log untuk reference ID tersebut. Kalau webhook memang gagal masuk, bisa trigger manual replay dari dashboard gateway. Kalau ini transaksi yang memang tidak valid, Agung proses refund manual ke user.
🟡
Jenis 2 — Ada di GoPop, tidak ada di Gateway (medium):
Transaksi tercatat SUCCESS di GoPop tapi gateway tidak settle dana-nya. Kemungkinan: transaksi di-capture di sisi GoPop tapi dibatalkan gateway karena fraud detection, atau ada timing issue.

Cara handle: Cek status transaksi di portal Midtrans/DOKU menggunakan transaction reference. Kalau gateway konfirmasi memang tidak settle, Agung perlu adjustment di pembukuan — dana ini tidak akan datang.
🟠
Jenis 3 — Ada di keduanya tapi nominal berbeda:
Biasanya karena perbedaan fee calculation atau ada partial refund yang dipotong di batch yang berbeda.

Cara handle: Bandingkan breakdown fee di laporan gateway dengan yang tercatat di sistem. Kalau ada partial refund, cek apakah sudah tercatat di rekonsiliasi. Kalau perbedaan kecil (<Rp1.000) bisa jadi rounding — catat sebagai adjustment.
2
Aturan Bisnis
▾
📌
Setiap selisih WAJIB diselesaikan dan didokumentasikan. Tidak boleh ada selisih yang di-"maafkan" atau diabaikan tanpa alasan tertulis. Semua resolusi dicatat di sistem: penyebab, aksi yang diambil, siapa yang memutuskan, dan kapan.
📌
Selisih lebih dari Rp500.000 wajib mendapat approval Agung sebelum ada adjustment apapun. Di bawah itu Rifky bisa handle mandiri tapi tetap harus dicatat.
📌
Rekonsiliasi yang masih punya selisih terbuka tidak bisa di-approve — harus semua diselesaikan dulu. Kalau ada selisih yang butuh waktu lebih dari 24 jam, tandai sebagai "Dalam Investigasi" dan lanjutkan rekonsiliasi periode lain. Selisih ini tetap harus diselesaikan, tidak boleh terlupakan.
3
Bad Cases
▾
Selisih sudah lebih dari 3 hari tidak terselesaikan dan penyebabnya tidak diketahui
—
Eskalasi ke Midtrans/DOKU melalui jalur resmi (email ke support mereka dengan bukti transaksi). Minta penjelasan tertulis dari gateway. Sementara menunggu, Agung buat "pending item" di pembukuan agar laporan keuangan tetap akurat sambil menunggu klarifikasi.
PIC: Agung (eskalasi formal ke gateway) · Rifky (siapkan bukti transaksi)
Selisih kumulatif dari beberapa periode menumpuk karena tidak diselesaikan tepat waktu
—
Ini tidak boleh terjadi. Kalau sudah terjadi, Rifky dan Agung perlu blokir waktu 1–2 hari khusus untuk membereskan semua selisih yang menumpuk dari yang terlama. Buat SOP yang jelas: kalau ada selisih yang tidak bisa diselesaikan dalam 24 jam, langsung eskalasi ke Acoy untuk bantu koordinasi.
PIC: Acoy (enforce SOP) · Rifky + Agung (prioritaskan resolusi)
4
Bottleneck
▾
Investigasi selisih butuh akses ke banyak sistem sekaligus — dashboard GoPop, portal Midtrans, portal DOKU, dan log server — dan sekarang harus dibuka satu per satu secara manual
Dashboard GoPop idealnya bisa langsung deep-link ke transaksi di portal gateway menggunakan reference ID. Kalau Rifky klik satu transaksi yang berselisih, ada tombol "Cek di Midtrans" atau "Cek di DOKU" yang langsung buka halaman transaksi tersebut di portal gateway. Ini memangkas waktu investigasi dari 30 menit per selisih jadi bisa 5 menit.
PIC: Isal + Ipong (deep-link ke portal gateway dari detail transaksi) · Irwan + Rifky (URL pattern masing-masing gateway)
5
Hasil Akhir
▾
✅ Selisih Terselesaikan
Penyebab selisih diidentifikasi dan didokumentasikan
Aksi korektif diambil (refund manual / adjustment pembukuan / replay webhook)
Rekonsiliasi bisa dilanjutkan dan di-approve
❌ Selisih Tidak Terselesaikan dalam 24 Jam
Tandai "Dalam Investigasi" → eskalasi formal ke gateway
Buat pending item di pembukuan Agung
Wajib diselesaikan sebelum periode tutup buku bulan ini

🗄️ Master Data › F07
Sync Master Data dari Digiflazz
Digiflazz adalah biller utama GoPop. Mereka punya ratusan produk (pulsa, paket data, token, voucher game, dll) yang terus berubah. Flow ini memungkinkan Ipong menarik daftar produk terbaru dari Digiflazz dan menyinkronkannya ke database GoPop tanpa input manual satu per satu.
👥 Ipong (Aggregator/Produk) 🔔 Digiflazz update produk, ada produk baru, atau harga berubah massal ⏱ Idealnya dijalankan otomatis harian — saat ini masih manual
1
Alur Normal
▾
1
Buka Master Data → Sync Digiflazz → tap "Sync Sekarang"
Sistem query API Digiflazz untuk pull daftar produk terbaru. Proses ini bisa memakan 30–60 detik tergantung jumlah produk. Progress bar tampil selama proses berlangsung.
2
Sistem bandingkan data dari Digiflazz vs database GoPop
Hasilnya tiga kategori perubahan:
  • Produk baru: Ada di Digiflazz tapi belum ada di GoPop — Ipong bisa pilih mana yang mau diaktifkan
  • Perubahan HPP: Produk yang sudah ada tapi HPP-nya berubah di Digiflazz — harga jual GoPop perlu dievaluasi ulang
  • Produk dihapus/nonaktif: Ada di GoPop tapi sudah tidak ada di Digiflazz — perlu dinonaktifkan di GoPop
3
Ipong review dan konfirmasi perubahan
Ipong tidak harus accept semua perubahan secara otomatis. Untuk produk baru, Ipong pilih mana yang mau diaktifkan dan set harga jual. Untuk perubahan HPP, Ipong review dan update harga jual supaya margin tetap positif. Untuk produk yang dihapus, nonaktifkan satu klik.
4
Apply perubahan — produk terupdate di app
Setelah Ipong konfirmasi, perubahan langsung berlaku. User yang cek produk berikutnya akan lihat daftar produk terbaru.
2
Aturan Bisnis
▾
📌
Sync tidak otomatis apply semua perubahan. Ipong harus review dan konfirmasi sebelum apply — terutama untuk perubahan HPP yang bisa mempengaruhi margin. Tidak ada auto-apply tanpa review manusia.
📌
Kalau HPP Digiflazz naik, Ipong WAJIB update harga jual GoPop agar margin tidak jadi negatif. Sistem akan alert Ipong kalau ada produk yang HPP-nya naik melebihi harga jual yang ada.
📌
Produk yang dihapus dari Digiflazz tapi masih ada transaksi yang sedang berjalan tidak bisa langsung dinonaktifkan — transaksi yang sudah PAID harus diselesaikan dulu (sukses atau gagal+refund), baru nonaktifkan produk.
📌
Sync sebaiknya dilakukan di luar jam sibuk (misal pagi sebelum jam 07.00 atau malam setelah jam 22.00) agar kalau ada masalah tidak langsung berdampak ke traffic tinggi.
3
Bad Cases
▾
Digiflazz naik HPP massal untuk pulsa semua operator — Ipong tidak sadar dan tidak update harga jual
—
Sistem harus alert Ipong dengan banner merah di dashboard kalau ada produk yang margin jadi negatif setelah sync HPP. Ipong tidak bisa klik "Apply" kalau masih ada produk dengan margin negatif — forced review. Akoy juga mendapat notifikasi kalau ada margin negatif yang belum ditangani lebih dari 1 jam.
PIC: Isal + Ipong (forced review sebelum apply kalau ada margin negatif) · Acoy (notif kalau tidak ditangani)
Sync gagal di tengah jalan — sebagian produk terupdate, sebagian tidak
—
Sync harus atomik — kalau gagal di tengah, rollback semua perubahan ke kondisi sebelumnya. Jangan biarkan database dalam state setengah-setengah. Tampilkan pesan error yang jelas ke Ipong dan log error untuk Irwan investigasi.
PIC: Irwan + Rifky (implementasi atomic sync dengan rollback) · Ipong (coba sync ulang setelah error)
4
Bottleneck
▾
Sync masih manual — Ipong harus ingat dan jalankan sendiri setiap hari. Kalau lupa, produk baru dari Digiflazz tidak tampil ke user
Idealnya ada scheduler yang otomatis pull data Digiflazz setiap malam jam 02.00 (saat traffic rendah), deteksi perubahan, dan kirim ringkasan ke Ipong via WA: "Ada 3 produk baru dan 2 perubahan HPP dari Digiflazz. Buka dashboard untuk review." Ipong tinggal review dan approve dari dashboard tanpa harus ingat untuk sync manual.
PIC: Irwan + Rifky (implementasi auto-scheduler sync Digiflazz + notif WA ke Ipong)
5
Hasil Akhir
▾
✅ Sync Sukses
Produk baru dari Digiflazz tersedia di app setelah Ipong review + apply
HPP terupdate — Ipong sudah sesuaikan harga jual
Produk yang discontinued sudah dinonaktifkan
❌ Risiko
HPP naik tidak terdeteksi → margin negatif → jual rugi
Sync gagal setengah jalan → database inconsistent
Lupa sync manual → produk baru tidak tersedia

Saldo & Limit Biller
Halaman paling kompleks dan paling kritis di Pengaturan Sistem. Menampilkan saldo prepaid GoPop di setiap supplier/biller (Supplier A, B, C, D), mengatur zonasi alert, dan mengontrol aksi otomatis saat saldo mendekati habis. Dari screenshot dashboard production yang terlihat, ini tampilan yang sudah berjalan live.
👥 Acoy (Super Admin) · Ipong (Aggregator — pantau rutin) 🔔 Saldo supplier mendekati batas alert / perlu isi ulang saldo ⏱ Dipantau real-time sepanjang jam operasional
1
Anatomi Halaman — dari Screenshot Production
▾
📸
Berdasarkan screenshot dashboard production, halaman Saldo & Limit Biller berisi empat card supplier secara horizontal. Berikut breakdown setiap elemen yang terlihat.
🟢
Status Saldo per Supplier — 4 Level:
  • ⚫ Blackout — Saldo habis atau sangat kritis, transaksi ke supplier ini tidak bisa diproses
  • 🔴 Danger — Saldo sangat rendah, perlu top-up segera dalam hitungan jam
  • 🟡 Warning — Saldo mulai menipis, perlu perhatian dan rencana top-up
  • 🟢 Aman — Saldo cukup untuk operasional normal
📊
Progress Bar Saldo: Visual bar di tiap card supplier menunjukkan proporsi saldo tersisa vs total kapasitas. Warna bar mengikuti status (hijau → kuning → merah → hitam).
🔢
Threshold Alert per Supplier (3 level): Setiap supplier punya tiga batas yang bisa diset Acoy:
  • Batas Danger (merah): misal Rp500.000 — saldo di bawah ini = status Danger
  • Batas Warning (kuning): misal Rp1.000.000 — saldo di bawah ini = status Warning
  • Batas Aman (hijau): misal Rp3.000.000 — saldo di atas ini = status Aman
Dari screenshot: Supplier A threshold Rp500k/Rp1jt/Rp3jt; Supplier B Rp500k/Rp1jt/Rp3jt; Supplier C rata-rata harian Rp1,8jt (Rp450k/Rp900k/Rp2,7jt); Supplier D Rp500k/Rp1jt/Rp3jt.
⚡
Toggle Manual / Otomatis per Supplier: Dua mode operasi:
  • Manual: Acoy/Ipong yang memutuskan kapan top-up saldo — sistem hanya alert tapi tidak aksi otomatis
  • Otomatis: Sistem otomatis top-up saldo ketika mencapai threshold Danger/Blackout, tanpa perlu intervensi manual. Butuh konfigurasi nominal top-up otomatis dan metode pembayaran yang sudah tersimpan.
Dari screenshot: Supplier D mode Otomatis aktif (toggle hijau). Supplier lain masih Manual. Ada field "Limit maks per transaksi saat kritis" untuk Supplier D (Rp200.000).
🔔
Aksi Pop-up per Status: Setiap supplier bisa dikonfigurasi untuk tampilkan pop-up ke user app/web saat saldo masuk status tertentu:
  • Toggle "Tampilkan pop-up saat Blackout" — default ON di semua supplier
  • Toggle "Tampilkan pop-up saat Danger" — bisa ON atau OFF per supplier
  • Toggle "Tampilkan pop-up saat Warning" — opsional
  • Toggle "Tampilkan pop-up saat Aman" — biasanya OFF (tidak perlu notif kalau aman)
Pop-up ini muncul di app user sebagai banner atau modal yang memberitahu layanan tertentu sedang gangguan/terbatas.
⚠️
Badge "Traffic Tinggi": Dari screenshot, Supplier D punya badge "⚠️ Traffic Tinggi" di pojok kanan atas card-nya. Ini indikator bahwa volume transaksi ke supplier ini sedang tinggi — kombinasi dengan saldo Blackout berarti situasi kritis yang perlu perhatian segera.
📢
Banner Info di Atas: Dari screenshot terlihat banner kuning yang menampilkan produk mana yang sedang aktif menampilkan pop-up ke end-user app:
  • "Danger: Supplier A — PDAM Surya Sembada Surabaya, Token Listrik PLN 100.000, Pulsa Telkomsel 10.000"
  • "Blackout: Supplier D — Pulsa Telkomsel 100.000, Token Listrik PLN 100.000, XL Hot Rod 12GB 30 Hari"
Ini penting agar Acoy/Ipong tahu persis produk mana yang sedang berdampak ke user.
2
Alur Normal — Top-up Saldo Supplier
▾
1
Acoy/Ipong melihat status saldo semua supplier
Buka Pengaturan Sistem → Saldo & Limit Biller. Langsung terlihat kondisi keempat supplier sekaligus. Card dengan status Danger atau Blackout tampil lebih mencolok (border merah, badge status besar).
2
Identifikasi supplier yang perlu top-up
Prioritaskan supplier dengan status Blackout terlebih dahulu (transaksi sudah tidak bisa jalan), lalu Danger (akan segera tidak bisa). Lihat juga badge "Traffic Tinggi" — kalau saldo rendah + traffic tinggi = situasi kritis yang perlu ditangani dalam hitungan menit.
3
Proses top-up saldo ke supplier
Top-up saldo ke Digiflazz/supplier dilakukan di luar dashboard GoPop — biasanya via transfer bank ke rekening supplier atau melalui portal supplier tersebut. Setelah transfer dikonfirmasi, saldo di dashboard akan terupdate (bisa otomatis via API atau manual refresh tergantung supplier).
4
Saldo terupdate — status kembali normal
Setelah saldo di-refresh, status card supplier berubah dari Blackout/Danger ke Warning atau Aman. Pop-up yang sebelumnya tampil ke user app otomatis menghilang (kalau toggle pop-up Aman = OFF). Banner info di atas halaman juga terupdate.
3
Alur — Konfigurasi Alert & Threshold
▾
1
Acoy set threshold per supplier
Klik tombol edit di card supplier → muncul form threshold: field Rp untuk Danger, Warning, dan batas Aman. Nilai threshold sebaiknya berdasarkan rata-rata pemakaian harian supplier tersebut. Dari screenshot Supplier C ada info "Rata-rata harian: Rp1.800.000" yang membantu Acoy menentukan threshold yang tepat.
2
Set toggle Manual atau Otomatis
Kalau supplier punya integrasi API top-up, set ke Otomatis dan konfigurasi nominal top-up otomatis (misal auto top-up Rp5.000.000 kalau saldo jatuh ke Danger). Kalau tidak ada API, tetap Manual dan pastikan Acoy/Ipong rutin pantau.
3
Set toggle pop-up per status per supplier
Tentukan status mana yang akan tampilkan pop-up ke user app. Rekomendasi: pop-up Blackout = SELALU ON (user harus tahu produk tidak bisa dipakai), pop-up Danger = ON untuk produk-produk kritis, pop-up Warning = opsional (bisa bikin user overthinking kalau terlalu sering).
4
Aturan Bisnis
▾
📌
Saldo supplier adalah uang GoPop yang diparkir di biller untuk memudahkan transaksi real-time. Ini bukan pendapatan — ini modal kerja. Terlalu sedikit = transaksi gagal. Terlalu banyak = modal kerja idle yang tidak efisien. Agung perlu terlibat dalam keputusan berapa saldo ideal per supplier.
📌
Pop-up ke user app adalah komunikasi krisis — jangan dipakai berlebihan. Kalau pop-up terlalu sering muncul (karena threshold Warning terlalu tinggi), user akan mulai mengabaikannya. Simpan pop-up untuk kondisi yang benar-benar berdampak ke user.
📌
Supplier D dengan mode Otomatis butuh pemantauan berbeda. Kalau auto top-up gagal (misal rekening sumber dana tidak ada, atau API supplier error), saldo bisa tetap di Blackout meski mode Otomatis aktif. Acoy perlu cek log auto top-up secara berkala.
📌
Badge "Traffic Tinggi" adalah early warning system. Kalau saldo Aman tapi traffic tinggi, saldo bisa cepat habis. Acoy/Ipong perlu proaktif top-up lebih awal dari biasanya saat traffic sedang tinggi (akhir bulan, tanggal gajian, event promo).
📌
"Auto-turunkan limit saat Danger/Blackout" — dari screenshot terlihat ada toggle ini. Artinya kalau saldo masuk Danger, sistem otomatis turunkan limit maksimum per transaksi ke nilai yang lebih kecil (misal dari Rp1.000.000 ke Rp200.000 seperti di Supplier D) untuk memperpanjang umur saldo yang tersisa sambil menunggu top-up.

Ini kondisi terburuk. Acoy turunkan threshold Warning ke nilai yang lebih realistis berdasarkan rata-rata konsumsi per jam (bukan per hari). Gunakan info "Rata-rata harian" yang sudah tampil di halaman ini untuk kalkulasi threshold yang tepat.
PIC: Acoy (kalibrasi ulang threshold) · Ipong (monitor dan rekomendasikan nilai berdasarkan pola traffic)

Minimal, ada shortcut link langsung ke portal supplier yang bersangkutan. Ini memangkas waktu response dari 15–30 menit jadi 2–5 menit saat kondisi kritis.

Konfigurasi Payment Gateway
Pengaturan koneksi ke Midtrans dan DOKU — API keys, webhook URL, mode (sandbox/production), dan metode pembayaran yang diaktifkan per gateway. Ini adalah konfigurasi teknikal sensitif yang jarang diubah kecuali ada perubahan kontrak atau rotasi API key.
👥 Acoy (Super Admin) · Irwan + Rifky (BE — implementasi) 🔔 Rotasi API key, perubahan kontrak gateway, atau onboarding metode bayar baru
1
Konfigurasi yang Tersedia per Gateway
▾
🔑
API Keys (Midtrans & DOKU): Server Key dan Client Key untuk production. Ditampilkan masked (****) dengan tombol "Salin" — tidak pernah tampil plaintext di dashboard. Ada tombol "Rotate Key" untuk memicu rotasi jika ada indikasi kebocoran.
🌐
Webhook URL: URL endpoint GoPop yang didaftarkan ke Midtrans/DOKU untuk menerima callback pembayaran. Kalau URL ini berubah (misal domain ganti), harus diupdate di portal Midtrans/DOKU dan di sini secara bersamaan. Webhook yang salah = transaksi tidak pernah terkonfirmasi.
🔄
Mode Sandbox / Production: Toggle untuk switch antara environment testing (sandbox) dan live (production). JANGAN aktifkan Sandbox di production — transaksi akan "sukses" di sistem tapi uang tidak benar-benar bergerak.
💳
Metode Pembayaran Aktif: Checklist metode yang diaktifkan per gateway — QRIS, VA BCA/BNI/Mandiri/BRI, E-wallet, Retail (Alfamart/Indomaret). Hanya aktifkan yang ada dalam kontrak dengan gateway.
⏱
Timeout VA: Durasi VA berlaku sebelum expired (default 24 jam). Bisa diubah per kebijakan bisnis — lebih pendek = user lebih terdorong bayar cepat, lebih panjang = lebih nyaman tapi dana floating lebih lama.
2
Aturan Bisnis
▾
📌
API Key TIDAK PERNAH boleh dicommit ke repository code. Harus selalu disimpan sebagai environment variable di server, bukan di config file yang masuk ke Git. Kalau API Key pernah terekspos di repository (bahkan sudah dihapus dari commit), anggap sudah bocor dan rotate segera.
📌
Setiap perubahan konfigurasi gateway harus ditest dulu di sandbox sebelum apply ke production. Kalau ada perubahan yang tidak bisa ditest di sandbox (misal ganti API Key production), lakukan di luar jam sibuk dan monitor webhook callback selama 30 menit setelahnya.
📌
Webhook signature verification wajib aktif. Midtrans dan DOKU mengirim signature di setiap callback untuk verifikasi bahwa callback benar-benar dari mereka. Kalau signature verification dinonaktifkan, sistem rentan terhadap serangan palsu callback yang bisa memicu transaksi sukses tanpa bayar.
3
Bad Cases
▾
API Key production terekspos atau dicurigai bocor
—
Langkah segera: (1) Rotate API Key baru dari portal Midtrans/DOKU, (2) Update di dashboard GoPop, (3) Monitor transaksi 24 jam ke belakang untuk deteksi transaksi unauthorized, (4) Lapor ke Midtrans/DOKU security team. Jangan tunda — setiap menit API Key bocor adalah risiko fraud.
PIC: Irwan (rotate key segera) · Acoy (update di dashboard + monitor) · Rifky (audit transaksi mencurigakan)
Webhook URL berubah (domain baru) tapi lupa update di portal Midtrans/DOKU — semua pembayaran tidak terkonfirmasi

Admin baru buka link di email, set password baru (wajib memenuhi syarat: minimal 8 karakter, ada angka dan huruf besar), konfirmasi password, klik "Aktifkan Akun". Status berubah dari "Invited" ke "Aktif" dan admin baru bisa langsung login.
4
Admin baru login pertama kali — lihat dashboard sesuai role
Dashboard yang tampil disesuaikan dengan role yang diassign. Admin Marketing hanya lihat menu CMS, Promo, dan SEO. Admin CS hanya lihat Komplain & Tiket dan Monitoring. Tidak ada konfigurasi tambahan yang diperlukan — role sudah handle semua.
5
Acoy brief admin baru tentang SOP dan akses yang dimiliki
Technical setup selesai, tapi Acoy tetap perlu brief admin baru tentang: SOP penggunaan dashboard, apa yang boleh dan tidak boleh dilakukan, cara eskalasi kalau ada masalah, dan siapa yang harus dihubungi untuk pertanyaan teknikal. Ini di luar sistem, tapi penting.

Aturan Bisnis
▾
📌
Link undangan berlaku 48 jam. Kalau admin baru belum aktivasi dalam 48 jam, Acoy perlu kirim ulang undangan. Link yang expired tidak bisa dipakai — ini mencegah link undangan yang terlupakan dipakai orang lain.
📌
Admin baru tidak bisa memilih role sendiri — role ditentukan Acoy saat membuat undangan. Kalau role perlu diubah setelah akun aktif, Acoy edit dari halaman detail akun admin tersebut.
📌
Satu undangan = satu email = satu akun. Tidak bisa kirim undangan ke email yang sudah punya akun aktif atau pending. Kalau perlu ubah email admin yang ada, suspend akun lama dan buat undangan baru.
📌
Akun admin baru langsung aktif tanpa perlu approval tambahan setelah admin set password. Tidak ada approval workflow — kepercayaan bahwa Acoy sudah memverifikasi identitas orang yang diundang sebelum mengirim undangan.

Bad Cases
▾
Admin baru salah email saat onboarding — undangan ke email yang salah
Admin baru tidak menerima email undangan.
Acoy batalkan undangan yang salah (dari halaman tim, admin dengan status "Invited" bisa di-cancel), lalu buat undangan baru dengan email yang benar. Tampilkan email tujuan undangan dengan jelas di halaman konfirmasi sebelum kirim, agar Acoy bisa double-check.
PIC: Acoy (batalkan + buat ulang undangan) · Isal + Ipong (tambah konfirmasi email sebelum kirim undangan)
Link undangan expired sebelum admin baru sempat aktivasi (lebih dari 48 jam)
Admin baru klik link → dapat pesan "Link undangan sudah kadaluarsa".
Acoy perlu kirim ulang undangan dari halaman tim — tombol "Kirim Ulang Undangan" di samping admin yang masih berstatus "Invited". Link baru fresh 48 jam. Admin baru tidak perlu melakukan apapun selain tunggu email baru.
