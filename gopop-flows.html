🔐 Login Admin › F01
Login & Masuk Dashboard
Titik masuk pertama semua anggota tim internal — dari Super Admin sampai CS. Login menggunakan email dan password, bukan nomor HP.
👥 Semua role admin 🔔 Buka gopop-lilac.vercel.app ⏱ <30 detik
1
Alur Normal
▾
1
Admin buka dashboard di browser
Ketik alamat dashboard di browser. Kalau belum pernah login atau sesi sudah habis, langsung diarahkan ke halaman login.
2
Masukkan email dan password
Admin isi email kerja dan password. Email bersifat unik per anggota tim — tidak bisa dua orang pakai email yang sama.
3
Klik "Masuk" — sistem verifikasi identitas
Sistem cek apakah email terdaftar dan password cocok. Kalau benar, sistem buat sesi login untuk admin ini.
4
Dashboard terbuka sesuai role
Admin masuk ke halaman utama dashboard. Menu yang muncul di sidebar berbeda-beda tergantung role:

  • Super Admin: Lihat dan akses semua menu
  • Ops (Acoy): Monitoring, Komplain, Reseller, Rekonsiliasi, Produk
  • Finance (Agung): Monitoring (read), Rekonsiliasi, Laporan
  • CS: Komplain, Pengguna (read-only), Monitoring (read-only)
  • IT: Semua menu untuk keperluan teknikal
2
Aturan Bisnis
▾
📌
Login admin menggunakan email, bukan nomor HP. Ini berbeda dari login consumer/reseller yang pakai nomor HP.
📌
Setelah 5 kali salah password dalam satu sesi, akun terkunci sementara selama 30 menit. Super Admin bisa unlock lebih cepat dari menu Role & Akses.
📌
Sesi login aktif selama 8 jam. Setelah itu, admin harus login ulang. Tidak ada opsi "ingat saya" untuk alasan keamanan.
📌
Satu akun admin tidak boleh dipakai bergantian oleh dua orang berbeda. Setiap anggota tim punya akun masing-masing agar semua aksi tercatat per individu.
3
Bad Cases
▾
Password salah berkali-kali — akun terkunci
"Akun kamu terkunci sementara. Coba lagi dalam 30 menit atau hubungi Super Admin untuk buka lebih cepat."
Super Admin buka dari menu Role & Akses → cari nama admin → klik "Unlock Akun". Atau tunggu 30 menit otomatis terbuka.
PIC: Super Admin (Acoy) untuk unlock manual
Admin lupa password
Klik "Lupa Password" di halaman login → isi email → cek email untuk link reset.
Link reset password dikirim ke email terdaftar, berlaku 1 jam. Kalau email tidak masuk, cek folder spam atau minta Super Admin reset manual dari dashboard.
PIC: Admin sendiri (via email) atau Super Admin (reset manual)
Admin baru tidak bisa login — akun belum dibuat
"Email tidak terdaftar."
Super Admin buat akun baru dari menu Role & Akses → Tim → Tambah Anggota. Isi email, tentukan role, dan kirim undangan. Admin baru dapat email untuk set password pertama kali.
PIC: Super Admin (Acoy)
4
Bottleneck
▾
Satu akun dipakai bergantian oleh beberapa orang — tidak ada jejak siapa yang melakukan apa
Tegaskan aturan: satu akun = satu orang. Kalau ada anggota baru, buatkan akun baru jangan pinjam akun lama. Ini penting karena semua aksi di dashboard (refund, suspend reseller, update status transaksi) dicatat atas nama akun yang login.
PIC: Acoy (pastikan SOP ini dijalankan)
Admin tidak logout saat meninggalkan komputer — risiko akses tidak sah
Sistem otomatis logout setelah tidak ada aktivitas selama 2 jam. Tapi lebih baik admin selalu logout manual saat meninggalkan meja. Tambahkan kebiasaan ini ke SOP onboarding tim.
PIC: Acoy (SOP tim)
5
Hasil Akhir
▾
✅ Flow Sukses
Admin masuk ke dashboard dengan menu sesuai role-nya
Semua aksi admin tercatat atas nama akun yang login
Sesi aktif 8 jam, setelah itu perlu login ulang
❌ Flow Gagal
Akun terkunci karena salah password 5x — perlu tunggu 30 menit atau minta Super Admin unlock
Email tidak terdaftar — perlu Super Admin buat akun dulu

Sesi Habis / Auto-Logout
Sesi login admin tidak berlangsung selamanya. Flow ini menjelaskan apa yang terjadi saat sesi habis dan bagaimana admin kembali masuk tanpa kehilangan pekerjaannya.
👥 Semua role admin 🔔 Sesi 8 jam habis atau idle 2 jam
1
Alur Normal
▾
1
Sistem deteksi sesi admin akan habis
15 menit sebelum sesi berakhir, muncul notifikasi kecil di pojok kanan atas: "Sesi kamu akan berakhir dalam 15 menit. Klik di sini untuk perpanjang."
2
Admin bisa perpanjang sesi
Klik notifikasi → sesi diperpanjang 8 jam lagi tanpa perlu input password ulang. Ini untuk menghindari admin terpaksa login ulang di tengah pekerjaan penting.
3
Kalau tidak diperpanjang — sesi habis
Dashboard otomatis redirect ke halaman login dengan pesan: "Sesi kamu sudah berakhir. Silakan masuk kembali untuk melanjutkan." Halaman yang sedang dibuka sebelumnya disimpan — setelah login, admin diarahkan kembali ke halaman yang sama.
4
Admin login ulang
Masukkan password → langsung kembali ke halaman yang sedang dikerjakan sebelum sesi habis.
2
Aturan Bisnis
▾
📌
Sesi aktif maksimal 8 jam sejak login terakhir, atau 2 jam sejak terakhir ada aktivitas (whichever comes first).
📌
Saat sesi habis di tengah pengisian form (misal sedang isi form refund), data yang belum disimpan akan hilang. Admin perlu isi ulang. Ini trade-off keamanan yang disengaja.
📌
Admin bisa logout manual kapan saja dari menu profil di pojok kanan atas. Disarankan selalu logout saat meninggalkan komputer.
3
Bad Cases
▾
Admin sedang proses refund besar — tiba-tiba sesi habis sebelum submit
Dashboard redirect ke login. Data refund yang sedang diisi hilang.
Admin login ulang → kembali ke halaman refund → isi ulang data. Ke depannya, sistem perlu simpan draft form secara otomatis setiap 30 detik untuk menghindari kehilangan data.
PIC: Isal + Ipong (implementasi auto-save draft)
4
Bottleneck
▾
Admin tidak tahu sesi akan habis — tiba-tiba ter-logout saat sedang kerja
Notifikasi 15 menit sebelum habis harus jelas dan tidak mudah terlewat. Kalau admin sedang fokus ke pekerjaan lain, notifikasi perlu ada suara atau visual yang mencolok bukan hanya teks kecil.
PIC: Isal + Ipong (perbaiki UI notifikasi sesi)
5
Hasil Akhir
▾
✅ Flow Sukses
Admin berhasil perpanjang sesi tanpa login ulang
Atau: login ulang dan kembali ke halaman yang sama
❌ Flow Gagal
Data form yang sedang diisi hilang karena sesi habis mendadak

Lihat & Filter Transaksi
Halaman utama Monitoring — tempat Rifky dan Acoy memantau semua transaksi yang masuk. Bisa disaring berdasarkan status, tanggal, produk, dan kata kunci.
👥 Rifky · Acoy · CS (read) 🔔 Setiap hari — pemantauan rutin
1
Alur Normal
▾
1
Buka menu Monitoring Transaksi
Klik menu "Monitoring" di sidebar. Halaman langsung menampilkan daftar transaksi terbaru, diurutkan dari yang paling baru di atas. Ada ringkasan angka di bagian atas: total transaksi hari ini, berapa yang sukses, gagal, dan sedang diproses.
2
Gunakan filter untuk mempersempit hasil
Tersedia filter:
  • Status: Semua / Menunggu Pembayaran / Diproses / Sukses / Gagal / Refunded / Dibatalkan / Perlu Review Manual
  • Tanggal: Hari ini / 7 hari / 30 hari / atau pilih rentang custom
  • Produk: Pulsa, Token Listrik, Transfer E-Wallet, dll
  • Pencarian: Ketik ID transaksi atau nomor HP user
3
Lihat daftar transaksi yang terfilter
Setiap baris menampilkan: waktu transaksi, nama user dan nomor HP, produk yang dibeli, nominal, status saat ini, dan nama supplier (biller) yang dipakai. Klik baris mana saja untuk lihat detail lengkapnya.
4
Export ke CSV kalau perlu
Klik tombol "Export CSV" di pojok kanan atas untuk download semua data yang sedang ditampilkan (sesuai filter aktif) ke dalam file Excel. Berguna untuk laporan harian Rifky.
2
Aturan Bisnis
▾
📌
CS hanya bisa lihat, tidak bisa aksi. CS tidak punya tombol refund, retry, atau update status. Mereka hanya bisa baca informasi untuk bantu jawab pertanyaan user.
📌
Status "Perlu Review Manual" artinya ada sesuatu yang tidak bisa diselesaikan otomatis oleh sistem — transaksi ini butuh perhatian Rifky atau Acoy segera.
📌
Data transaksi tersedia real-time. Tidak perlu refresh manual — halaman update otomatis setiap 30 detik selama halaman dibuka.
3
Bad Cases
▾
Filter tidak menghasilkan transaksi yang dicari — ID transaksi tidak ditemukan
Tabel kosong dengan pesan "Tidak ada transaksi yang cocok."
Pastikan ID transaksi yang diketik benar (bisa cek dari screenshot atau WA yang diterima user). Coba juga cari dengan nomor HP user sebagai alternatif. Kalau masih tidak ketemu, mungkin transaksi belum selesai dibuat (status draft).
PIC: CS atau Rifky yang sedang investigasi
Terlalu banyak transaksi "Perlu Review Manual" menumpuk
—
Ini tanda ada masalah di sisi supplier atau payment gateway. Rifky dan Acoy perlu segera cek satu per satu. Kalau jumlahnya banyak sekaligus, kemungkinan ada gangguan massal dari satu supplier — perlu koordinasi dengan Ipong untuk switch ke supplier backup.
PIC: Rifky + Acoy (investigasi) · Ipong (koordinasi supplier)
4
Bottleneck
▾
Tidak ada tab khusus untuk transaksi yang butuh tindakan segera — tercampur dengan semua transaksi
Perlu ditambahkan tab "Perlu Perhatian" yang otomatis menampilkan: transaksi berstatus "Perlu Review Manual", transaksi yang sudah "Diproses" lebih dari 15 menit tanpa update, dan transaksi "Gagal" yang belum di-refund. Rifky tidak perlu filter manual setiap kali — tinggal klik satu tab.
PIC: Isal + Ipong (tambah tab khusus di Monitoring)
5
Hasil Akhir
▾
✅ Flow Sukses
Tim punya gambaran jelas kondisi transaksi hari ini
Transaksi bermasalah teridentifikasi dan bisa segera ditangani
Export CSV tersedia untuk laporan
❌ Flow Gagal
Transaksi yang dicari tidak ditemukan karena ID salah atau data belum sync

📊 Monitoring Transaksi › F02
Lihat Detail Transaksi
Setelah menemukan transaksi yang dicari, admin buka detail lengkapnya. Di sini terlihat seluruh riwayat apa yang terjadi pada transaksi tersebut dari awal sampai akhir.
👥 Rifky · Acoy · CS 🔔 Klik baris transaksi di Monitoring
1
Alur Normal
▾
1
Klik baris transaksi — halaman detail terbuka
Halaman detail menampilkan semua informasi transaksi dalam satu tampilan.
2
Baca informasi transaksi
Informasi yang tersedia:
  • Identitas: ID transaksi, waktu dibuat
  • User: Nama, nomor HP, tipe akun (consumer/reseller/guest)
  • Produk: Nama produk, nominal, tujuan (nomor HP/ID pelanggan)
  • Pembayaran: Metode bayar, jumlah yang dibayar, waktu bayar
  • Supplier: Nama biller yang dipakai, referensi dari supplier
  • Status: Status saat ini beserta waktu terakhir berubah
3
Lihat timeline riwayat transaksi
Di bawah info utama ada timeline yang menampilkan setiap perubahan status dari awal sampai sekarang. Contoh: 14:30 Transaksi dibuat → 14:31 Pembayaran diterima → 14:31 Dikirim ke supplier → 14:32 Sukses.
Kalau ada admin yang pernah lakukan aksi manual (retry, refund, update status), juga tercatat di sini beserta nama adminnya.
2
Aturan Bisnis
▾
📌
Detail transaksi bersifat read-only untuk CS. Mereka bisa lihat semua info tapi tidak ada tombol aksi. Tombol aksi (refund, retry, update status) hanya muncul untuk Ops dan Super Admin.
📌
Timeline riwayat tidak bisa dimanipulasi. Semua yang tercatat di timeline adalah fakta permanen — berguna sebagai bukti kalau ada dispute dengan user atau supplier.
📌
Kalau transaksi punya tiket komplain terkait, ada link langsung ke tiket tersebut di halaman detail. CS tidak perlu cari manual di menu Komplain.
3
Bad Cases
▾
Timeline kosong atau tidak lengkap — tidak ada riwayat yang bisa dibaca
—
Ini tanda ada masalah teknikal dalam pencatatan riwayat transaksi. Rifky perlu catat ID transaksi yang bermasalah dan laporkan ke Irwan/Rifky BE untuk investigasi. Sementara itu, status transaksi masih bisa dibaca dari field status utama.
PIC: Irwan + Rifky (investigasi pencatatan timeline)
4
Bottleneck
▾
CS perlu tahu detail transaksi tapi tidak punya akses — harus minta ke Rifky setiap saat
CS seharusnya bisa lihat detail transaksi (read-only) tanpa perlu koordinasi. Akses read-only ke Monitoring sudah cukup — CS tidak perlu bisa aksi apapun. Ini mengurangi beban Rifky dari pertanyaan yang bisa dijawab sendiri oleh CS.
PIC: Acoy (pastikan role CS punya akses read-only ke Monitoring)
5
Hasil Akhir
▾
✅ Flow Sukses
Admin punya gambaran lengkap apa yang terjadi pada transaksi ini
Timeline lengkap tersedia sebagai bukti audit
Link ke tiket komplain terkait mudah diakses
❌ Flow Gagal
Timeline kosong — informasi riwayat tidak bisa dibaca

F03: Handle Transaksi Bermasalah →
📊 Monitoring Transaksi › F03
Handle Transaksi Bermasalah
Transaksi dengan status "Perlu Review Manual" atau yang stuck terlalu lama di "Diproses" perlu ditangani secara manual oleh Acoy atau Rifky. Flow ini menjelaskan langkah-langkahnya.
👥 Acoy · Rifky 🔔 Transaksi stuck / Perlu Review Manual ⏱ Target selesai dalam 1 jam kerja
1
Alur Normal
▾
1
Identifikasi transaksi yang bermasalah
Buka tab "Perlu Perhatian" di Monitoring — semua transaksi yang butuh tindakan sudah terkumpul di sini. Atau cari spesifik dengan filter status "Perlu Review Manual".
2
Buka detail dan pahami situasinya
Baca timeline transaksi. Lihat di mana tepatnya macet: apakah macet di pembayaran, di pengiriman ke supplier, atau supplier tidak memberi konfirmasi? Informasi ini menentukan aksi apa yang tepat.
3
Pilih aksi yang tepat
Ada beberapa pilihan aksi tergantung situasi:
  • Cek Status ke Supplier: Tanya langsung ke sistem supplier apakah transaksi ini sukses atau gagal di sisi mereka
  • Tandai Sukses Manual: Kalau sudah konfirmasi dari supplier bahwa produk terkirim tapi sistem belum update — gunakan ini. Wajib isi alasan.
  • Tandai Gagal & Proses Refund: Kalau sudah dipastikan gagal — ubah status ke gagal dan langsung proses refund ke user
4
Isi alasan aksi — submit
Setiap aksi manual wajib disertai alasan singkat (minimal 10 kata). Ini penting untuk audit trail. Contoh: "Konfirmasi dari Digiflazz via email bahwa pulsa sudah terkirim jam 14:35, referensi #DGF-88821."
5
Sistem catat aksi dan update status
Status transaksi berubah. Aksi Rifky/Acoy tercatat di timeline beserta waktu dan nama. Kalau ada tiket komplain terkait transaksi ini, status tiket juga otomatis diupdate.
2
Aturan Bisnis
▾
📌
Wajib konfirmasi ke supplier sebelum tandai sukses manual. Jangan pernah tandai transaksi sukses hanya berdasarkan asumsi — harus ada bukti dari supplier bahwa produk benar-benar terkirim.
📌
Aksi "Cek Status ke Supplier" bisa dilakukan maksimal 3 kali dalam 1 jam untuk transaksi yang sama. Lebih dari itu berarti ada masalah serius di supplier yang perlu Ipong tangani langsung.
📌
Kalau memilih "Tandai Gagal", refund akan diproses otomatis jika user sudah membayar. Tidak perlu proses refund terpisah.
📌
Semua aksi manual dicatat permanen. Tidak bisa dihapus atau diubah setelah disubmit. Pastikan yakin sebelum klik submit.
3
Bad Cases
▾
Rifky tandai sukses manual — ternyata produk tidak terkirim ke user
User WA ke CS: "Pulsa saya belum masuk tapi status sudah Sukses."
Ini situasi kritis. CS eskalasi ke Acoy segera. Acoy perlu hubungi supplier langsung untuk klarifikasi. Kalau terbukti produk tidak terkirim, perlu koordinasi dengan Agung untuk proses kompensasi manual ke user. Timeline di sistem akan tercatat semua perubahan ini sebagai bukti.
PIC: Acoy (koordinasi) · Agung (kompensasi) · Ipong (klarifikasi ke supplier)
Transaksi sudah ditandai selesai tapi refund tidak bisa diproses karena nominal tidak cocok
—
Ini terjadi kalau ada selisih antara nominal yang dibayar user dan nominal di sistem. Agung perlu investigasi manual dan proses refund via metode alternatif sambil koordinasi dengan Irwan untuk investigasi penyebab selisih.
PIC: Agung + Irwan + Rifky
4
Bottleneck
▾
Tidak ada standar berapa lama transaksi boleh stuck sebelum perlu diintervensi manual
Perlu ada SLA yang jelas: transaksi "Diproses" lebih dari 15 menit otomatis masuk ke tab "Perlu Perhatian" dan Rifky mendapat notifikasi. Tanpa standar ini, transaksi bisa stuck berhari-hari tanpa ada yang handle.
PIC: Acoy (tetapkan SLA) · Irwan + Rifky (implementasi notifikasi otomatis)
Rifky tidak tahu harus hubungi siapa di supplier untuk klarifikasi cepat
Buat dan simpan kontak PIC dari setiap supplier (Digiflazz) di tempat yang mudah diakses tim — bisa di halaman Produk & Harga atau dokumen internal. Jangan sampai waktu terbuang hanya untuk cari kontak saat situasi darurat.
PIC: Ipong (maintain kontak supplier)
5
Hasil Akhir
▾
✅ Flow Sukses
Transaksi berhasil diklarifikasi dan status diupdate dengan tepat
Aksi tercatat lengkap di timeline sebagai audit trail
User mendapat kepastian — sukses atau refund diproses
❌ Flow Gagal
Tandai sukses tapi produk tidak terkirim — user komplain
Supplier tidak responsif — status tidak bisa dikonfirmasi

📊 Monitoring Transaksi › F04
Proses Refund Manual
Refund manual dilakukan ketika refund otomatis gagal atau tidak berjalan. Ini adalah proses yang melibatkan Agung (Finance) karena menyangkut pengembalian dana ke user.
👥 Agung (Finance) · Acoy (approve) 🔔 Refund otomatis gagal atau transaksi ditandai Gagal ⏱ Target: Dana kembali dalam 1×24 jam
1
Alur Normal
▾
1
Identifikasi transaksi yang perlu refund
Bisa berasal dari dua jalur: (1) CS eskalasi tiket komplain yang butuh refund, atau (2) Rifky/Acoy menemukan transaksi gagal yang belum di-refund saat monitoring rutin.
2
Buka detail transaksi — klik "Proses Refund"
Tombol "Proses Refund" hanya muncul kalau: (1) transaksi statusnya Gagal, dan (2) user sudah membayar (status pembayaran bukan Expired atau Dibatalkan). Kalau tidak memenuhi syarat, tombol tidak muncul.
3
Isi detail refund dan submit
Form refund berisi:
  • Nominal refund: Terisi otomatis sesuai jumlah yang dibayar user. Agung bisa ubah kalau ada alasan khusus (refund parsial).
  • Alasan refund: Wajib diisi. Contoh: "Transaksi gagal dari sisi supplier Digiflazz, produk tidak terkirim."
4
Sistem proses refund via payment gateway
Sistem kirim permintaan refund ke Midtrans/DOKU sesuai metode bayar asal. Status transaksi berubah ke "Refunded". WA otomatis dikirim ke user: "Dana Rp [nominal] sedang kami kembalikan ke [metode bayar]. Estimasi masuk 1–3 hari kerja."
5
Monitor konfirmasi refund selesai
Beberapa jam kemudian, payment gateway mengirim konfirmasi bahwa refund sudah selesai. Status transaksi berubah final. WA kedua dikirim ke user: "Dana sudah kembali ke akun kamu."
2
Aturan Bisnis
▾
📌
GoPop tidak memotong biaya apapun saat refund. User mendapat kembali 100% jumlah yang dibayar. Biaya admin yang sudah dibayar ke payment gateway tidak ditagihkan ke user.
📌
Refund hanya bisa diproses ke metode bayar yang sama dengan yang dipakai saat transaksi. Tidak bisa refund ke rekening berbeda atau metode berbeda.
📌
Refund untuk transaksi yang sudah berstatus Sukses tidak bisa diproses dari dashboard biasa — butuh persetujuan Super Admin dan prosedur khusus karena berarti ada dispute tentang kualitas produk.
📌
Satu transaksi hanya bisa di-refund satu kali. Sistem akan tolak percobaan refund kedua untuk transaksi yang sama.
3
Bad Cases
▾
Refund gagal di sisi payment gateway — Midtrans/DOKU menolak permintaan
User belum menerima dana kembali meski sudah dijanjikan.
Agung cek alasan penolakan dari log dashboard. Biasanya karena akun user sudah tidak aktif atau metode bayar sudah kadaluarsa. Agung perlu hubungi user via WA untuk cari solusi alternatif (transfer manual ke rekening user). Koordinasi dengan Irwan untuk investigasi kenapa gateway menolak.
PIC: Agung (handle ke user) · Irwan (investigasi gateway)
User claim refund tidak masuk setelah 3 hari kerja
User WA CS: "Katanya sudah direfund tapi tidak masuk ke OVO saya."
CS cek di detail transaksi apakah ada konfirmasi refund selesai dari gateway. Kalau ada konfirmasi, berarti masalah di sisi user (cek saldo OVO, cek di akun yang terdaftar). Kalau tidak ada konfirmasi, eskalasi ke Agung untuk investigasi ke Midtrans/DOKU langsung.
PIC: CS (cek status) → Agung (eskalasi ke gateway kalau perlu)
4
Bottleneck
▾
Agung tidak selalu online — refund terlambat diproses saat Agung sedang tidak di dashboard
Perlu ada mekanisme notifikasi WA ke Agung saat ada refund yang perlu diproses — terutama kalau nominalnya besar atau sudah menunggu lebih dari 2 jam. Untuk refund di bawah nominal tertentu (misalnya di bawah Rp 100.000), Acoy bisa diberikan akses untuk proses tanpa perlu tunggu Agung.
PIC: Acoy (tentukan threshold) · Irwan + Rifky (implementasi notifikasi)
5
Hasil Akhir
▾
✅ Flow Sukses
Dana kembali 100% ke metode bayar asal user dalam 1–3 hari kerja
User menerima WA konfirmasi di dua tahap: saat diproses dan saat selesai
Status transaksi berubah ke Refunded — tercatat permanen
❌ Flow Gagal
Gateway menolak refund — perlu solusi alternatif manual
User claim tidak menerima dana setelah 3 hari — perlu investigasi lanjut

User Submit Tiket dari Pusat Bantuan
Flow dimulai dari sisi user — mereka punya masalah dan ingin melapor. Bisa dilakukan lewat app, web, atau reseller app. Tidak perlu login — guest pun bisa komplain.
👥 User · Guest · Reseller 🔔 User punya masalah dengan transaksi atau akun ⏱ 2–3 menit pengisian form
1
Alur Normal
▾
1
User buka Pusat Bantuan
Dari menu navigasi bawah atau dari shortcut di screen transaksi gagal.
2
Isi form komplain
Tiga field utama yang wajib diisi:
  • Nomor WhatsApp Aktif — nomor yang bisa dihubungi CS. Kalau sudah login, terisi otomatis dari profil.
  • Kategori Masalah — pilih dari dropdown: Transaksi Gagal, Saldo Tidak Masuk, Masalah Akun/Login, Lainnya
  • Deskripsi — ceritakan masalahnya dengan kata-kata sendiri
Ada satu field opsional: ID Transaksi — kalau user ingat dan mau cantumkan.
3
Tap "Kirim Laporan"
Tombol tidak bisa diklik dua kali — mencegah tiket duplikat. Loading sebentar (1–2 detik) saat sistem memproses.
4
Tiket dibuat — user dapat konfirmasi
Dua konfirmasi sekaligus:
  • Di app/web: Screen "Laporan Terkirim" dengan nomor tiket, misalnya #TKT-20240628-001, dan estimasi respons dalam 1×24 jam.
  • Via WhatsApp: Pesan otomatis ke nomor yang diisi user berisi nomor tiket dan instruksi simpan nomor tiket.
2
Aturan Bisnis
▾
📌
Nomor WhatsApp adalah satu-satunya cara CS menghubungi user. Tidak ada balasan via app atau email — semua komunikasi CS ke user melalui WA ke nomor yang diisi di form.
📌
Tiket yang dibuat guest (tanpa login) tetap valid dan mendapat penanganan yang sama. Nomor WA menjadi identifikasi utama guest.
📌
Sistem otomatis tentukan prioritas tiket berdasarkan kategori: "Transaksi Gagal" dapat prioritas Tinggi, kategori lain dapat prioritas Normal.
📌
Tiket yang sama dari user yang sama dalam waktu kurang dari 30 menit akan ditolak sistem sebagai duplikat.
3
Bad Cases
▾
User isi nomor WA yang salah — CS tidak bisa menghubungi
User tidak mendapat WA konfirmasi dan tidak ada yang menghubungi dalam 24 jam.
User perlu submit tiket baru dengan nomor WA yang benar. Sayangnya tidak ada cara edit tiket yang sudah submit. Ke depan, tambahkan langkah konfirmasi OTP ringan saat isi nomor WA agar user tahu nomornya benar sebelum submit.
PIC: Bili (pertimbangkan tambah verifikasi nomor WA) · Agung + Fikly (implementasi FE)
WA konfirmasi tidak terkirim karena layanan WA belum terintegrasi
Tiket berhasil dibuat dan nomor tiket muncul di screen, tapi tidak ada WA yang masuk.
Sementara ini, pastikan screen konfirmasi di app menampilkan nomor tiket dengan jelas agar user tetap punya bukti. Sampaikan ke user (via onboarding atau FAQ) bahwa konfirmasi WA mungkin sedikit terlambat. Prioritaskan implementasi integrasi WA.
PIC: Irwan + Rifky (integrasi WA Business API)
4
Bottleneck
▾
User tidak tahu status tiket mereka setelah submit — harus tunggu WA dari CS
Tambahkan halaman "Riwayat Tiket" di app/web yang bisa diakses tanpa login (cukup dengan nomor tiket + nomor WA). User bisa cek sendiri apakah tiket sudah diproses, sedang diproses, atau sudah selesai — tanpa harus menunggu dihubungi CS.
PIC: Bili (desain screen) · Agung + Fikly (implementasi FE)
5
Hasil Akhir
▾
✅ Flow Sukses
Tiket tersimpan di sistem dengan nomor unik
CS on-duty otomatis menerima notifikasi tiket baru
User mendapat nomor tiket sebagai bukti laporan diterima
❌ Flow Gagal
Nomor WA yang diisi salah — CS tidak bisa follow up
WA konfirmasi tidak terkirim karena layanan WA belum ready

CS Terima & Review Tiket Baru
Tiket yang baru masuk perlu segera diterima dan di-review oleh CS. Flow ini dimulai dari sisi dashboard — CS mendapat notifikasi dan membuka tiket untuk dipahami.
👥 CS · Acoy (monitor) 🔔 Tiket baru masuk dari user ⏱ Target review pertama: <1 jam
1
Alur Normal
▾
1
Sistem assign tiket ke CS on-duty
Tiket otomatis di-assign ke CS yang sedang aktif (online di dashboard). Sistem distribusi bergiliran — tidak selalu ke CS yang sama agar beban merata.
2
CS mendapat notifikasi di dashboard
Badge merah di menu Komplain bertambah. Ada pop-up kecil di pojok kanan bawah: "📥 Tiket baru: Transaksi Gagal dari +62812xxxx". Tiket prioritas Tinggi punya notifikasi yang lebih mencolok.
3
CS buka list tiket — cari yang belum ditangani
Di halaman Komplain & Tiket, CS bisa filter: "Milik Saya" atau "Belum Ditangani". Tiket baru muncul di bagian paling atas dengan label hijau "BARU". Tiket prioritas Tinggi ada label merah.
4
CS buka detail tiket dan baca semua informasi
Panel detail menampilkan: deskripsi masalah user, kategori, nomor WA user, dan kalau ada ID transaksi — otomatis muncul ringkasan transaksi tersebut (produk, nominal, status, waktu). CS tidak perlu pindah ke menu Monitoring untuk cek transaksi terkait.
5
CS tap "Mulai Proses" — tiket terkunci ke CS ini
Status tiket berubah dari Baru ke Sedang Diproses. Nama CS tercatat sebagai PIC. CS lain bisa lihat tiket tapi tidak bisa ambil alih kecuali dilepas oleh CS ini atau oleh Acoy.
2
Aturan Bisnis
▾
📌
Tiket yang belum dibuka dalam 30 menit setelah masuk akan di-reassign ke CS lain yang sedang aktif. Kalau semua CS offline, Acoy mendapat notifikasi sebagai monitor.
📌
Satu tiket hanya bisa aktif di satu CS sekaligus. Tidak bisa dua CS handle tiket yang sama bersamaan — saat CS pertama klik "Mulai Proses", tiket terkunci untuk mereka.
📌
Waktu "pertama kali dibuka" dicatat otomatis untuk keperluan laporan SLA response time.
3
Bad Cases
▾
Tiket tidak memiliki ID transaksi — CS tidak tahu kasus mana yang dimaksud user
—
CS gunakan fitur "Cari Transaksi" di dalam detail tiket: masukkan nomor HP user atau perkiraan waktu kejadian → sistem tampilkan kandidat transaksi. CS pilih yang paling relevan dan link ke tiket ini.
PIC: CS (investigasi manual) · Isal + Ipong (pastikan fitur cari transaksi di detail tiket tersedia)
CS sibuk — tiket menumpuk tidak ada yang handle
User tidak mendapat respons dalam 24 jam.
Acoy sebagai monitor harus pantau dashboard Komplain setiap hari untuk pastikan tidak ada tiket yang terlalu lama "Baru" tanpa diambil CS. Kalau beban terlalu berat, Acoy bisa re-assign tiket ke CS yang lebih available atau handle sendiri yang urgent.
PIC: Acoy (monitor distribusi beban CS)
4
Bottleneck
▾
Auto-assign tidak bekerja kalau semua CS offline di luar jam kerja — tiket menunggu tidak ada yang tangani
Tentukan jam operasional CS yang jelas dan sampaikan ke user (misalnya 08.00–22.00 WIB). Di luar jam operasional, tiket yang masuk tetap diterima tapi responnya di hari/shift berikutnya. Tambahkan pesan otomatis WA ke user: "Terima kasih, kami akan merespons pada jam operasional berikutnya (besok jam 08.00 WIB)."
PIC: Acoy (tentukan jam operasional) · Irwan + Rifky (implementasi auto-reply WA di luar jam operasional)
5
Hasil Akhir
▾
✅ Flow Sukses
CS paham masalah user dan siap untuk investigate
Tiket berstatus "Sedang Diproses" dengan PIC yang jelas
Response time pertama tercatat untuk laporan SLA
❌ Flow Gagal
Tiket tidak diambil CS dalam 30 menit — user menunggu tanpa kabar

CS Investigate & Selesaikan Tiket
CS sudah baca tiket dan mulai proses. Sekarang CS perlu investigate masalahnya, hubungi user via WA, dan selesaikan tiket dengan resolusi yang tepat.
👥 CS🔔 Tiket status "Sedang Diproses"⏱ Target selesai: <24 jam
1
Alur Normal
▾
1
CS cek status transaksi terkait
Klik "Lihat Detail Transaksi" di dalam tiket → langsung ke halaman detail transaksi di Monitoring. CS baca timeline: di mana macetnya, sudah berapa kali retry, apakah refund sudah berjalan.
2
Hubungi user via WhatsApp
Klik tombol "Hubungi via WA" → nomor WA user langsung terbuka. CS pilih template pesan yang sesuai situasi:
  • "Dana kamu aman, sedang kami proses. Estimasi selesai [waktu]."
  • "Transaksi kamu sudah sukses — [produk] sudah terkirim ke [tujuan]."
  • "Kami perlu eskalasi ke tim terkait. Estimasi 2×24 jam."
Setelah menghubungi, klik "Tandai Sudah Dihubungi" — waktu kontak dicatat.
3
Kalau masalah bisa diselesaikan sendiri — tutup tiket
Klik "Tutup Tiket" → pilih jenis resolusi (wajib):
  • Transaksi Berhasil
  • Refund Diproses
  • Edukasi User
  • Tidak Ada Masalah
  • Lainnya
Tambahkan catatan penutupan → submit. Status berubah ke Selesai.
4
Kalau tidak bisa diselesaikan sendiri — eskalasi (lihat F04)
Lanjut ke flow eskalasi kalau butuh akses atau wewenang yang CS tidak punya.
2
Aturan Bisnis
▾
📌
CS wajib menghubungi user via WA dalam 1 jam pertama setelah ambil tiket. Tidak perlu langsung ada solusi — cukup konfirmasi bahwa laporan sudah diterima dan sedang diproses.
📌
Kalau user tidak balas WA dalam 3×24 jam, CS boleh tutup tiket dengan resolusi "Tidak Ada Respons dari User" setelah kirim WA reminder terakhir.
📌
CS tidak boleh menjanjikan refund ke user sebelum mendapat konfirmasi dari Agung (Finance). Cukup sampaikan bahwa sedang diproses.
3
Bad Cases
▾
User marah dan mengancam akan lapor ke OJK/media sosial
User semakin emosional meski sudah dihubungi CS.
CS tetap profesional, tidak membalas emosi. Sampaikan bahwa masalah sedang ditangani dengan serius dan berikan estimasi waktu yang jelas. Eskalasi ke Acoy untuk di-handle langsung oleh yang lebih senior. Kalau ada ancaman serius ke reputasi GoPop, Acoy koordinasi dengan Bili.
PIC: Acoy (handle langsung) · Bili (keputusan komunikasi publik)
CS tidak tahu cara menjawab pertanyaan teknikal tentang kenapa transaksi gagal
CS memberikan jawaban yang tidak akurat dan user semakin bingung.
CS cukup sampaikan: "Kami sedang investigasi lebih lanjut dan akan update kamu segera." Jangan tebak-tebak penyebab teknikal. Eskalasi ke Rifky/Acoy untuk jawaban yang akurat, baru sampaikan ke user.
PIC: Acoy (buat knowledge base jawaban standar untuk CS)
4
Bottleneck
▾
CS tidak punya panduan jawaban standar — setiap CS menjawab berbeda untuk kasus yang sama
Buat knowledge base sederhana di dashboard: daftar skenario umum (pulsa gagal, token listrik tidak masuk, dll) beserta template jawaban yang sudah disetujui. CS tinggal pilih template yang sesuai dan modifikasi sedikit sesuai konteks. Ini memastikan konsistensi jawaban dan mempercepat response time.
PIC: Acoy (buat knowledge base awal) · Isal + Ipong (integrasikan ke dashboard)
5
Hasil Akhir
▾
✅ Flow Sukses
Tiket ditutup dengan resolusi yang jelas
User sudah dihubungi dan tahu statusnya
Durasi penanganan tercatat untuk laporan SLA
❌ Flow Gagal
Masalah tidak bisa diselesaikan CS sendiri → eskalasi ke F04
User tidak balas WA dalam 3 hari → tutup dengan resolusi "Tidak Ada Respons"

Eskalasi Tiket ke Ops atau Finance
Ada kasus yang CS tidak bisa selesaikan sendiri — butuh akses atau wewenang lebih tinggi. Flow ini menjelaskan cara eskalasi yang tepat dan apa yang terjadi setelahnya.
👥 CS → Ops (Acoy) / Finance (Agung)🔔 CS tidak bisa resolve sendiri⏱ Target respons eskalasi: <2 jam
1
Kapan Harus Eskalasi
▾
⚙️
Eskalasi ke Ops (Acoy): Butuh cek/retry transaksi ke supplier, force update status transaksi, atau investigasi transaksi stuck lebih dari 1 jam.
💰
Eskalasi ke Finance (Agung): Butuh proses refund manual, approve refund di atas batas nominal CS, atau investigasi selisih saldo reseller.
2
Alur Normal
▾
1
CS tap "Eskalasi Tiket" di detail tiket
Muncul form eskalasi: pilih tujuan (Ops/Finance/IT), isi alasan eskalasi (wajib, minimal 20 kata), pilih prioritas (Normal atau Urgent).
2
Notifikasi dikirim ke yang bersangkutan
Acoy atau Agung dapat notifikasi di dashboard dan WA internal: nama CS yang eskalasi, kategori masalah, alasan eskalasi, dan prioritas. Status tiket berubah ke Dieskala.
3
Acoy/Agung buka tiket dan ambil tindakan
Mereka bisa lihat semua riwayat tiket dan percakapan sebelumnya. Lakukan aksi yang diperlukan langsung dari dashboard (retry, refund, update status) tanpa koordinasi manual.
4
Selesai — update kembali ke CS
Acoy/Agung klik "Selesai dari Sisi Saya" + isi catatan aksi yang dilakukan. Tiket kembali ke CS dengan status "Perlu Follow-up ke User". CS dapat notifikasi dan langsung WA user dengan update.
3
Aturan Bisnis
▾
📌
Eskalasi hanya bisa dilakukan satu kali per tiket per level. CS → Ops/Finance. Tidak bisa CS → Ops → Finance → Ops lagi. Kalau perlu koordinasi antar Ops dan Finance, lakukan via catatan internal di tiket.
📌
Eskalasi Urgent yang tidak direspons dalam 1 jam akan trigger pengingat WA kedua ke yang bersangkutan, dan notifikasi ke Acoy sebagai monitor.
📌
Semua catatan eskalasi, aksi yang dilakukan, dan hasilnya tersimpan permanen di timeline tiket sebagai audit trail.
4
Bad Cases
▾
Acoy dan Agung keduanya tidak available — eskalasi tidak direspons
User tidak mendapat update setelah dijanjikan 2 jam.
CS informasikan ke user bahwa penanganan sedikit terlambat karena tim sedang handle prioritas lain, berikan estimasi baru. Jangan biarkan user tanpa update lebih dari 4 jam. Ke depan, tentukan backup PIC untuk eskalasi kalau kedua orang utama tidak available.
PIC: Acoy (tentukan backup eskalasi)
5
Bottleneck
▾
Eskalasi yang tidak perlu — CS eskalasi hal-hal yang seharusnya bisa diselesaikan sendiri
Buat panduan yang jelas: apa saja yang CS bisa selesaikan sendiri vs yang harus dieskalasi. Terlalu sering eskalasi hal kecil akan membebani Acoy dan Agung. Sebaliknya, terlalu jarang eskalasi bisa menyebabkan masalah besar tidak tertangani. Knowledge base yang baik akan mengurangi eskalasi yang tidak perlu.
PIC: Acoy (buat panduan eskalasi yang jelas)
6
Hasil Akhir
▾
✅ Flow Sukses
Masalah yang butuh akses lebih tinggi tertangani dengan benar
Semua aksi tercatat di timeline — audit trail lengkap
User mendapat update via WA dengan solusi konkret
❌ Flow Gagal
Tidak ada yang merespons eskalasi — user menunggu tanpa kabar
Ping-pong eskalasi bolak-balik — user tidak dapat kepastian

🎫 Komplain & Tiket › F05
Tiket Re-open Setelah Ditutup
Kadang tiket sudah ditutup tapi masalah belum benar-benar selesai. User bisa re-open dalam 3 hari setelah tiket ditutup — semua riwayat sebelumnya tetap ada.
👥 User + CS🔔 User masalah belum selesai setelah tiket ditutup⏱ Window re-open: 3 hari setelah tutup
1
Alur Normal
▾
1
User minta buka kembali tiket
Dua cara:
  • Balas WA terakhir dari CS dalam 3 hari → sistem deteksi balasan → tiket otomatis re-open
  • Dari app: Buka Riwayat Tiket → pilih tiket yang "Selesai" → tap "Masalah Belum Selesai? Buka Kembali" → isi alasan singkat
2
Tiket re-open dengan semua riwayat sebelumnya
Label RE-OPENED muncul di tiket. CS yang sama otomatis di-assign ulang. Ada section baru "Alasan Re-open" di atas tiket. Semua thread sebelumnya masih terbaca di bawah.
3
CS handle ulang dengan konteks lengkap
CS baca alasan re-open → cek ulang status transaksi terkait → hubungi user dengan update terbaru. Kalau masalahnya berbeda dari sebelumnya → bisa eskalasi lagi (F04).
2
Aturan Bisnis
▾
📌
Re-open hanya bisa dilakukan dalam 3 hari setelah tiket ditutup. Lebih dari itu, user perlu buat tiket baru (bisa cantumkan nomor tiket lama sebagai referensi).
📌
Satu tiket bisa di-reopen maksimal 2 kali. Re-open ketiga otomatis di-flag ke Acoy sebagai monitor — berarti ada sesuatu yang perlu perhatian khusus.
📌
Re-open tidak akan diterima kalau status akhir tiket adalah "Tidak Ada Respons dari User" — user yang tidak responsif tidak bisa tiba-tiba re-open setelah 3 hari. Harus buat tiket baru.
3
Bad Cases
▾
User re-open tiket berkali-kali padahal masalah sudah selesai — hanya ingin perhatian lebih
—
Setelah re-open ke-2, Acoy di-cc secara otomatis dan bisa lihat polanya. Kalau terbukti masalah sudah selesai (ada bukti di timeline), Acoy bisa tutup tiket secara final dan blokir re-open lebih lanjut untuk tiket ini.
PIC: Acoy (monitor dan putuskan)
4
Bottleneck
▾
CS berbeda yang handle re-open — tidak tahu konteks sebelumnya
Sistem prioritaskan assign re-open ke CS yang sama. Kalau CS itu tidak aktif, ada banner peringatan di atas tiket untuk CS pengganti: "Tiket ini sebelumnya ditangani oleh [nama]. Baca riwayat lengkap sebelum hubungi user." Seluruh riwayat selalu tersedia di bawah.
PIC: Irwan + Rifky (preferential assign) · Isal + Ipong (banner konteks di UI)
5
Hasil Akhir
▾
✅ Flow Sukses
User punya jalur follow-up tanpa buat tiket baru dari awal
CS punya konteks lengkap dari interaksi sebelumnya
Masalah akhirnya benar-benar selesai dengan konfirmasi dari user
❌ Flow Gagal
Sudah lewat 3 hari — user harus buat tiket baru
Re-open berulang tanpa penyelesaian nyata — perlu Acoy intervensi

Approve / Reject Reseller Baru
Setiap orang yang mendaftar sebagai reseller tidak langsung aktif — perlu persetujuan admin dulu. Flow ini menjelaskan proses review dan keputusan approve atau reject.
👥 Acoy (Ops)🔔 Ada pendaftar reseller baru⏱ Target: Keputusan dalam 1×24 jam kerja
1
Alur Normal
▾

Dua jalur deteksi: (1) Rifky lapor ada banyak transaksi gagal dari supplier yang sama saat monitoring rutin, atau (2) sistem auto-alert ke Ipong dan Acoy saat error rate satu supplier melebihi batas normal dalam 15 menit terakhir.
2
Ipong konfirmasi gangguan ke Digiflazz
Login ke portal Digiflazz atau hubungi kontak PIC Digiflazz yang sudah tersimpan. Konfirmasi apakah ada gangguan dari sisi mereka dan estimasi waktu recovery. Sambil menunggu konfirmasi, mulai persiapan switch.
3
Switch ke supplier backup di dashboard
Di halaman detail produk yang terdampak, ubah urutan prioritas supplier: geser supplier backup ke posisi pertama, supplier utama ke posisi kedua. Klik "Simpan" → transaksi berikutnya otomatis ke supplier backup.
4
Koordinasi update banner ke Kayong
Kalau gangguan diperkirakan lebih dari 30 menit, minta Kayong pasang banner di homepage: "Layanan [nama produk/operator] sedang gangguan. Kami sedang menangani." Ini mengurangi transaksi gagal dan tiket komplain yang masuk.
5
Monitor dan switch balik saat supplier utama recover
Saat Digiflazz sudah stabil, switch kembali ke konfigurasi semula. Minta Kayong hapus banner maintenance dari CMS.
2
Aturan Bisnis
▾
📌
HPP di supplier backup mungkin berbeda dengan Digiflazz. Kalau HPP backup lebih mahal, margin GoPop lebih kecil selama periode gangguan — ini trade-off yang diterima demi tetap melayani user.
📌
Transaksi yang sudah dikirim ke Digiflazz dan masih "Diproses" saat gangguan terjadi tidak otomatis di-retry ke backup — perlu ditangani manual oleh Rifky/Acoy satu per satu di Monitoring.
📌
Setiap switch supplier dicatat otomatis: siapa yang switch, dari mana ke mana, dan kapan. Berguna untuk evaluasi performa dan reliabilitas supplier.
3
Bad Cases
▾
Tidak ada supplier backup yang dikonfigurasi — tidak bisa switch ke mana-mana
Semua transaksi produk terdampak gagal sampai Digiflazz recover.
Nonaktifkan produk terdampak sementara agar user tidak bisa checkout (dan uang tidak tertahan). Pasang banner maintenance. Ini harus jadi pelajaran untuk segera setup supplier backup sebelum gangguan berikutnya terjadi.
PIC: Ipong (setup supplier backup sebagai prioritas) · Kayong (banner maintenance)
Digiflazz sudah recover tapi Ipong lupa switch balik — semua transaksi terus ke backup yang lebih mahal
—
Ipong set alarm/reminder manual saat switch ke backup. Ke depan bisa ada fitur auto-switch balik saat Digiflazz terdeteksi stabil kembali selama X menit berturut-turut.
PIC: Ipong (reminder manual) · Irwan + Rifky (auto-switch fitur)
4
Bottleneck
▾
Tidak ada auto-alert saat error rate Digiflazz tinggi — Ipong tahu gangguan hanya dari laporan Rifky atau user komplain
Sistem harus pantau error rate per supplier secara real-time. Kalau lebih dari X% transaksi ke satu supplier gagal dalam 15 menit, kirim alert WA otomatis ke Ipong dan Acoy. Deteksi lebih cepat = dampak lebih kecil ke user.
PIC: Irwan + Rifky (implementasi auto-monitoring error rate supplier)
5
Hasil Akhir
▾
✅ Flow Sukses
Layanan tetap berjalan via supplier backup selama gangguan
User hanya sedikit yang terdampak
Switch tercatat untuk evaluasi supplier
❌ Flow Gagal
Tidak ada backup → layanan mati total selama gangguan
Lupa switch balik → margin lebih kecil dari seharusnya

👤 Pengguna › F02
Handle Akun User yang Bermasalah
Akun user bisa bermasalah karena berbagai alasan: terkunci karena salah password berkali-kali, belum terverifikasi OTP, atau ter-suspend. CS perlu tahu cara menangani setiap situasi ini dengan benar.
👥 CS · Acoy (untuk suspend/reaktivasi)🔔 User komplain tidak bisa login
1
Tiga Situasi & Cara Handle
▾
🔒
Akun Terkunci (salah password 5x):
CS klik "Buka Kunci Akun" di profil user → akun langsung bisa dipakai tanpa menunggu 30 menit. Wajib verifikasi identitas dulu — tanya data yang hanya pemilik asli yang tahu (nama lengkap, nomor HP terdaftar, atau tanggal bergabung kira-kira).
📱
Akun Belum Terverifikasi (belum OTP sejak daftar):
CS klik "Kirim Ulang OTP Verifikasi" → user dapat WA dengan kode OTP baru yang berlaku 5 menit. Berguna kalau user bilang OTP-nya tidak datang saat pertama daftar.
🚫
Akun Ter-suspend:
CS tidak bisa lakukan reaktivasi — ini wewenang Acoy atau Super Admin. CS informasikan ke user untuk hubungi GoPop dengan alasan yang jelas kenapa ingin reaktivasi. CS buat tiket dan eskalasi ke Acoy.
2
Aturan Bisnis
▾
📌
Verifikasi identitas WAJIB sebelum unlock akun. CS tidak boleh unlock akun tanpa konfirmasi bahwa yang minta adalah pemilik asli — ada risiko orang lain mencoba ambil alih akun user lain.
📌
Semua aksi CS pada akun user dicatat di sistem: siapa CS yang melakukan, aksi apa, dan kapan. Ini penting kalau ada dispute atau audit keamanan.
📌
CS tidak bisa melakukan suspend akun user — itu wewenang Acoy dan Super Admin saja.
3
Bad Cases
▾
Orang lain yang mengaku sebagai pemilik akun minta CS unlock — identitas mencurigakan
—
CS tolak dengan sopan dan minta user verifikasi melalui jalur yang lebih aman: minta reset password via OTP ke nomor HP yang terdaftar. Kalau bisa melakukan itu, berarti mereka punya akses ke nomor HP pemilik asli. Jangan pernah unlock berdasarkan "feeling" — selalu verifikasi data konkret.
PIC: CS (keputusan kasus per kasus) · Acoy (eskalasi kalau ada indikasi pengambilalihan akun)
OTP sudah dikirim ulang berkali-kali tapi tidak masuk ke WA user
"Kode OTP tidak pernah datang ke WA saya."
CS perlu troubleshoot bersama user: (1) pastikan nomor HP yang terdaftar benar, (2) pastikan WA di nomor tersebut aktif dan tidak ada masalah, (3) minta user tunggu 5 menit lalu coba lagi. Kalau masih tidak masuk setelah 3 percobaan, kemungkinan ada masalah di layanan pengiriman WA — eskalasi ke Irwan untuk investigasi teknikal.
PIC: CS (troubleshoot awal) · Irwan (investigasi teknikal layanan WA)
4
Bottleneck
▾
Status "Terkunci" dan "Belum Terverifikasi" tidak tampil di dashboard — CS tidak tahu kondisi akun sebenarnya
Status akun harus lengkap dan jelas di profil user: bukan hanya Aktif/Suspend, tapi juga Terkunci (beserta sisa waktu buka otomatis) dan Belum Terverifikasi. Tanpa ini CS tidak bisa bantu user yang datang dengan masalah login tanpa tahu apa penyebabnya.
PIC: Isal + Ipong (tampilkan semua status akun) · Irwan + Rifky (pastikan data status tersedia di BE)
5
Hasil Akhir
▾
✅ Flow Sukses
Akun terkunci berhasil dibuka — user bisa login kembali
OTP terkirim — user bisa verifikasi akun
Akun suspend → tiket dibuat dan dieskalasi ke Acoy
❌ Flow Gagal
Identitas tidak terverifikasi — CS tidak bisa unlock, user perlu jalur lain
OTP tidak terdelivery — masalah di layanan WA perlu investigasi Irwan

Kelola Role & Permission Tim
Acoy sebagai Super Admin mengatur apa yang bisa dilihat dan dilakukan setiap anggota tim di dashboard. Ini memastikan tidak ada yang bisa mengakses hal yang bukan wewenangnya.
👥 Super Admin (Acoy)🔔 Ada kebutuhan perubahan permission role
1
Alur Normal
▾
1
Buka menu Role & Akses
Tampil semua role yang ada: Super Admin, Ops, Finance, CS, IT. Setiap role punya deskripsi singkat siapa yang memakainya dan berapa anggota tim yang punya role tersebut saat ini.
2
Pilih role yang akan diubah
Klik nama role → muncul matrix permission: daftar semua menu dan aksi di dashboard dengan toggle aktif/nonaktif. Contoh: role CS punya toggle aktif untuk "Lihat Monitoring" tapi nonaktif untuk "Aksi di Monitoring" seperti refund atau retry.
3
Update permission — simpan
Toggle permission yang perlu diubah → klik "Simpan Perubahan". Berlaku langsung untuk semua anggota tim dengan role tersebut — tidak perlu mereka logout/login ulang.
2
Aturan Bisnis
▾
📌
Hanya Super Admin yang bisa mengubah permission role. Tidak ada role lain yang bisa masuk ke menu ini — mencegah admin memberi dirinya sendiri akses lebih dari yang seharusnya.
📌
Permission yang diset di sini harus di-enforce juga di server, bukan hanya menyembunyikan menu di tampilan. Seseorang yang tahu URL endpoint tidak boleh bisa bypass pembatasan ini.
📌
Setiap perubahan permission dicatat permanen: siapa Acoy yang mengubah, apa yang diubah, dan kapan. Tidak bisa dihapus dari log.
3
Bad Cases
▾
Permission terlalu luas — CS bisa akses dan aksi hal yang bukan wewenangnya (misal refund)
—
Audit log menampilkan aksi yang tidak seharusnya. Acoy perbaiki permission CS segera dan review semua aksi yang dilakukan CS dalam periode permission terlalu luas. Kalau ada aksi finansial yang tidak seharusnya, Agung perlu investigasi dampaknya.
PIC: Acoy (perbaiki permission + audit dampak) · Agung (dampak finansial)
Permission hanya di tampilan (FE) tapi tidak di server — ada yang bypass lewat URL langsung
—
Ini gap keamanan serius (D-GAP-21). Setiap endpoint sensitif di server HARUS punya pengecekan permission sendiri — tidak boleh hanya mengandalkan tampilan yang disembunyikan di FE. Harus diimplementasi Irwan + Rifky di BE sebagai prioritas tinggi.
PIC: Irwan + Rifky (permission check di setiap endpoint BE)
4
Bottleneck
▾
Acoy tidak bisa verifikasi permission sudah benar tanpa login pakai akun role lain untuk cek
Tambahkan fitur "Preview sebagai Role X" — Acoy bisa lihat tampilan dashboard persis seperti yang dilihat CS atau Finance tanpa perlu buat akun baru atau logout. Ini sangat membantu saat setup permission pertama kali atau setelah ada perubahan besar.
PIC: Isal + Ipong (implementasi preview mode per role)
5
Hasil Akhir
▾
✅ Flow Sukses
Setiap anggota tim hanya bisa akses dan aksi sesuai wewenangnya
Perubahan permission berlaku langsung tanpa perlu logout ulang
❌ Flow Gagal
Permission terlalu luas — ada yang bisa aksi di luar wewenang
Permission hanya di FE, tidak di server — bisa di-bypass

Tambah & Nonaktifkan Anggota Tim
Saat ada anggota tim baru yang perlu akses dashboard, atau anggota lama yang sudah tidak bekerja di GoPop, Acoy yang mengatur akses mereka. Ini harus dilakukan segera — jangan tunggu berhari-hari.

1
Alur Tambah Anggota Baru
▾
1
Buka menu Tim di Role & Akses
Tampil semua anggota tim yang punya akses dashboard, beserta role dan status akun masing-masing (Aktif/Nonaktif).
2
Klik "Tambah Anggota" — isi data
Isi: email kerja anggota baru, nama lengkap, dan pilih role yang sesuai dengan posisi mereka. Email ini yang akan dipakai untuk login ke dashboard.
3
Sistem kirim email undangan
Anggota baru dapat email berisi link untuk set password pertama kali. Link berlaku 24 jam. Setelah set password, mereka langsung bisa login dengan role yang sudah ditentukan Acoy.
2
Alur Nonaktifkan Anggota yang Keluar
▾
1
Temukan anggota di daftar Tim
2
Klik "Nonaktifkan Akun" — konfirmasi
Akun langsung tidak bisa dipakai login. Kalau yang bersangkutan sedang login saat dinonaktifkan, mereka otomatis di-logout dalam beberapa menit. Semua riwayat aksi mereka tetap tersimpan — tidak bisa dihapus.
⚠️
Penting: Nonaktifkan akun anggota yang keluar sesegera mungkin — jangan tunggu. Akun aktif mantan karyawan adalah risiko keamanan serius yang tidak boleh dibiarkan.
3
Aturan Bisnis
▾
📌
Akun dinonaktifkan, bukan dihapus. Semua riwayat aksi mantan anggota tetap ada sebagai bagian dari audit trail. Nama mereka masih muncul di log aksi lama.
📌
Satu email hanya bisa dipakai satu akun. Kalau seseorang bergabung kembali setelah keluar, akun lama bisa diaktifkan kembali — tidak perlu buat akun baru.
📌
Acoy disarankan melakukan audit daftar anggota setiap bulan — bandingkan dengan daftar karyawan aktif yang sebenarnya. Nonaktifkan yang sudah tidak relevan.
4
Bad Cases
▾
Mantan karyawan masih punya akses aktif karena lupa dinonaktifkan
—
Acoy audit daftar akun aktif vs daftar karyawan aktif yang sebenarnya. Nonaktifkan semua yang sudah tidak bekerja. Kalau ditemukan akses mencurigakan dari akun yang harusnya nonaktif, investigasi segera apakah ada data yang diakses atau aksi yang dilakukan.
PIC: Acoy (audit bulanan dan tindakan segera)
Link undangan anggota baru sudah expired sebelum dibuka — sudah lewat 24 jam
Anggota baru tidak bisa set password karena link kadaluarsa.
Acoy klik "Kirim Ulang Undangan" di profil anggota yang bersangkutan. Link baru dikirim ke email yang sama. Sarankan anggota baru langsung set password begitu email masuk — jangan ditunda.
PIC: Acoy (kirim ulang undangan)
5
Hasil Akhir
▾
✅ Tambah Anggota
Anggota baru dapat email undangan dan bisa set password
Setelah set password, langsung bisa akses dashboard sesuai role
✅ Nonaktifkan Anggota
Akun langsung tidak bisa dipakai login
Riwayat aksi tetap tersimpan untuk audit
