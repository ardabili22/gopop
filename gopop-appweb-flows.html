App & Web Consumer › Modul 01 › F01
Load Homepage
Flow yang terjadi setiap kali user atau guest membuka app GoPop atau mengunjungi gopop.id untuk pertama kali, atau setelah lama tidak dibuka. Homepage menampilkan greeting, shortcut produk, banner promo, dan artikel Tips & Edukasi.
🎨
Referensi Figma
File: bwdy1MflFyrW5S7vvIDTy0 · 390px mobile portrait
00 - Homepage (1:5452)
👤 User Login · Guest 🔔 Buka app atau kunjungi gopop.id ⏱ Target load <2 detik
1
Alur Normal
▾
1
App terbuka — langsung tampil skeleton loading
Sambil data dimuat dari server, app langsung tampilkan layout homepage tapi dengan konten abu-abu berkedip (skeleton). Ini supaya user tahu app tidak hang — sedang loading. Navbar bawah sudah bisa diklik selama loading.
2
Sistem cek status login user
App cek apakah ada sesi login aktif di penyimpanan lokal. Hasilnya menentukan tampilan header:

Kalau sudah login: Header menampilkan "HAI, [NAMA USER]! 👋" + foto profil kecil di pojok kanan atas.
Kalau guest/belum login: Header menampilkan "Selamat datang di GoPop 👋" + tombol "Masuk / Daftar".
3
Data konten dimuat dari server secara bersamaan
Dua permintaan data dikirim sekaligus agar lebih cepat:
  • Konten CMS: Banner promo aktif dan artikel Tips & Edukasi dari dashboard Kayong/Luqman
  • Produk populer: 8 produk untuk ditampilkan di grid shortcut
Biasanya selesai dalam 1–2 detik kalau koneksi normal.
4
Konten tampil — skeleton digantikan konten nyata
Berurutan dari atas ke bawah, konten muncul menggantikan skeleton:
  • Greeting + sub-judul "Mau bayar apa hari ini?"
  • Card produk — filter chip (Semua/Isi Ulang/Tagihan) + grid 2 baris × 4 kolom: Transfer, Pulsa, Token PLN, Paket Data, BPJS, Game, E-Money, Lainnya
  • Strip CTA Transfer E-Wallet — "GoPay → OVO • Fee Rp 1.000" dengan tombol "Coba →"
  • Section "Promo Spesial 🔥" — banner carousel horizontal, swipe kiri-kanan
  • Section "Tips & Edukasi" — 2 artikel terbaru dari CMS
5
Homepage siap digunakan — user bisa langsung aksi
Semua elemen interaktif aktif: tap produk, swipe banner, scroll ke bawah, tap artikel, atau tap menu di navbar bawah (Beranda, Layanan, Transfer, Riwayat, Profil).
2
Aturan Bisnis
▾
📌
Homepage bisa diakses tanpa login (guest-first). Semua konten homepage tampil penuh untuk guest — tidak ada konten yang disembunyikan atau dikunci. Guest bisa langsung tap produk dan transaksi.
📌
Konten CMS di-cache selama 5 menit. Kalau user buka homepage lagi dalam 5 menit, app tampilkan data cache dulu sambil refresh di background. Ini membuat homepage terasa lebih cepat saat re-open.
📌
Strip CTA Transfer E-Wallet selalu tampil. Ini adalah hero product GoPop dengan fee Rp 1.000 flat. Strip ini tidak bisa dinonaktifkan dari CMS — selalu ada di homepage di bawah grid produk.
📌
Banner carousel otomatis berganti tiap 3 detik. User tetap bisa swipe manual untuk berganti banner. Kalau tidak ada banner aktif dari CMS, section "Promo Spesial" tidak tampil — tidak ada placeholder kosong.
📌
Data Tips & Edukasi juga dari CMS Kayong. Hanya artikel dengan status "Published" yang tampil. Tampilkan maksimal 2 artikel di homepage — tap "Selengkapnya" untuk ke halaman listing penuh (screen 29).
3
Bad Cases
▾
Server tidak merespons — data konten tidak dimuat sama sekali
Yang user lihat: Skeleton terus berkedip lebih dari 5 detik, tidak ada konten yang muncul.
Setelah 5 detik timeout, tampilkan homepage versi minimal: grid 8 produk yang di-hardcode (tidak dari server), tanpa banner dan tanpa artikel. User masih bisa tap produk dan lanjut transaksi. Tampilkan pesan kecil di bagian atas: "Koneksi lambat. Menampilkan konten tersimpan."
PIC: Agung (App) + Fikly (Desktop) — implementasi fallback konten hardcode
Gambar banner dari CMS gagal dimuat — gambar rusak atau ukuran salah
Yang user lihat: Area banner menampilkan kotak kosong atau ikon gambar rusak.
Tampilkan placeholder berwarna ungu (#4A2D8C) dengan logo GoPop kecil di tengah sebagai fallback. Banner tetap bisa diklik ke tujuan yang benar meski gambarnya tidak muncul. Kayong perlu compress gambar ke bawah 2MB dan pastikan format JPG/PNG saat upload ulang.
PIC: Agung + Fikly (implementasi image fallback) · Kayong (compress gambar yang bermasalah)
Sesi login expired tapi user tidak tahu — halaman tetap tampil greeting nama tapi API calls return unauthorized
Yang user lihat: Homepage tampil normal dengan nama mereka, tapi beberapa aksi tidak bisa dilakukan.
Saat ada respons "tidak terautentikasi" dari server, app otomatis coba refresh sesi di background. Kalau berhasil, user tidak perlu tahu. Kalau gagal, bersihkan data sesi dan tampilkan homepage versi guest dengan notifikasi kecil: "Sesi kamu habis. Masuk lagi untuk transaksi penuh."
PIC: Agung + Fikly (implementasi auto-refresh sesi dan graceful degradation)
CMS belum terhubung ke server — banner yang Kayong ubah tidak pernah muncul ke user
Yang user lihat: Banner tidak berubah meski Kayong sudah update di dashboard.
Ini gap kritis (D-GAP-18). Irwan + Rifky perlu menghubungkan CMS dashboard ke server sebagai prioritas tinggi. Sementara belum tersambung, homepage menampilkan banner dan artikel dummy yang di-hardcode di FE.
PIC: Irwan + Rifky (hubungkan CMS ke BE) · Agung + Fikly (hardcode konten sementara)
4
Bottleneck
▾
13+ versi "00 - Homepage" di Figma belum dikunci — Agung dan Fikly tidak bisa mulai implement
Ini blocker terbesar saat ini. Agung dan Fikly tidak tahu versi mana yang harus diimplementasi. Bili perlu pilih 1 versi final, rename jadi "00 - Homepage [FINAL]", dan arsipkan sisanya. Estimasi cukup 30 menit untuk Bili melakukan ini.
🎨 Bili: Action item — kunci versi final Homepage secepatnya
Load homepage lambat kalau koneksi user sedang buruk — user menunggu terlalu lama
Implementasi strategi "tampil dulu, update belakangan": selalu tampilkan data dari cache lokal dulu (meski mungkin sudah 5 menit lalu), sambil ambil data baru di background. User langsung lihat konten, bukan loading spinner. Hanya pertama kali install yang perlu loading penuh karena belum ada cache.
PIC: Agung + Fikly (implementasi cache strategy di FE)
5
Hasil Akhir
▾
✅ Flow Sukses
Homepage tampil lengkap dalam <2 detik
Greeting sesuai status login (nama atau "Selamat datang")
Banner dari CMS Kayong tampil dan bisa diklik
Grid 8 produk shortcut siap untuk di-tap
Artikel Tips & Edukasi terbaru tampil
❌ Flow Gagal / Degraded
Server timeout → homepage minimal dengan produk hardcode
Gambar banner rusak → placeholder ungu dengan logo
CMS belum terhubung → banner dummy permanen sampai D-GAP-18 selesai

Filter Produk di Homepage
Di dalam card produk homepage ada tiga filter chip: Semua, Isi Ulang, dan Tagihan. User bisa tap untuk menyaring produk yang ditampilkan di grid sesuai kategori yang mereka butuhkan.
🎨
Referensi Figma
Filter chip terlihat di dalam card produk homepage. Chip aktif: background ungu #4A2D8C, teks putih. Chip nonaktif: background putih, border abu.
00 - Homepage (1:5452) — section Filter Chips
👤 User Login · Guest 🔔 User tap chip filter di card produk ⏱ Instan — tidak ada loading
1
Alur Normal
▾
1
Default — chip "Semua" aktif saat homepage pertama dibuka
Grid menampilkan 8 produk campuran: Transfer, Pulsa, Token PLN, Paket Data, BPJS, Game, E-Money, Lainnya. Chip "Semua" tampil dengan background ungu (aktif), dua chip lain tampil dengan background putih dan border (nonaktif).
2
User tap chip "Isi Ulang"
Chip "Isi Ulang" langsung berubah jadi aktif (ungu), chip "Semua" kembali ke nonaktif. Grid produk berubah menampilkan hanya produk kategori isi ulang: Pulsa, Paket Data, Token PLN, Voucher Game. Tidak ada loading — perubahan instan karena data sudah ada di app.
3
User tap chip "Tagihan"
Grid berubah menampilkan hanya produk tagihan/postpaid: BPJS Kesehatan, BPJS Ketenagakerjaan, Internet & TV, PDAM, PBB, Listrik Pascabayar. User bisa scroll horizontal di dalam card kalau produk lebih dari 8.
4
User tap produk dari hasil filter — langsung ke screen produk tersebut
Filter tidak memengaruhi flow setelah tap produk. Setelah user masuk ke screen produk dan kembali ke homepage, filter kembali ke "Semua" (tidak mengingat pilihan terakhir).
2
Aturan Bisnis
▾
📌
Filter berjalan sepenuhnya di sisi app (offline). Tidak ada request ke server saat user tap chip filter — semua produk sudah dimuat saat homepage pertama dibuka. Ini membuat filter terasa instan.
📌
Strip CTA Transfer E-Wallet selalu tampil di bawah card produk, tidak peduli filter apa yang aktif. Transfer E-Wallet adalah hero product GoPop dan selalu diprioritaskan.
📌
Filter tidak disimpan saat user keluar dari homepage. Setiap kali buka homepage, filter selalu kembali ke "Semua". Ini keputusan UX yang disengaja agar user selalu melihat semua opsi dulu.
3
Bad Cases
▾
Setelah filter "Tagihan" aktif, user tap "Lainnya" yang tidak ada di kategori ini — tapi produk itu tetap perlu bisa diakses
Yang user lihat: Chip "Tagihan" aktif, grid hanya menampilkan produk tagihan — "Lainnya" tidak ada.
User bisa tap chip "Semua" untuk melihat semua produk termasuk "Lainnya". Atau tap "Layanan" di navbar bawah untuk ke screen 28 yang menampilkan semua produk tanpa filter. Pastikan ada tombol "Lihat Semua Layanan" yang selalu terlihat di bawah grid, apapun filter yang aktif.
PIC: Bili (konfirmasi apakah perlu tombol "Lihat Semua" di bawah grid)
Filter "Tagihan" menghasilkan lebih dari 8 produk — grid tidak cukup tampung semua
Yang user lihat: Hanya 8 produk pertama yang tampil, sisanya tersembunyi.
Grid homepage maksimal 8 produk (2 baris × 4). Kalau produk di kategori lebih dari 8, tampilkan tombol "Lihat semua [kategori]" di bawah grid yang mengarah ke screen 28 Layanan Semua dengan filter yang sudah terisi. User tidak kehilangan akses ke produk yang tidak tampil.
PIC: Bili (desain tombol "Lihat semua" per kategori) · Agung + Fikly (routing ke screen 28 dengan filter)
4
Bottleneck
▾
Kategori produk di filter chip belum terdefinisi secara formal — Agung dan Fikly tidak tahu produk mana masuk ke "Isi Ulang" dan mana ke "Tagihan"
Bili perlu buat daftar mapping yang jelas: produk mana saja yang masuk ke kategori "Isi Ulang" dan "Tagihan". Ini harus disepakati bersama Acoy dan Ipong (yang kelola produk di dashboard) agar konsisten antara FE dan BE. Tanpa mapping yang jelas, Agung dan Fikly bisa salah implement.
PIC: Bili (buat mapping produk per kategori) · Acoy + Ipong (validasi dari sisi bisnis)
5
Hasil Akhir
▾
✅ Flow Sukses
Grid produk berubah instan sesuai filter yang dipilih
Strip Transfer E-Wallet tetap tampil apapun filternya
User mudah menemukan produk yang dibutuhkan
❌ Flow Gagal
Produk tidak muncul di kategori yang benar karena mapping belum terdefinisi
Grid penuh 8 produk — produk lain tersembunyi tanpa ada akses

F03: Tap Shortcut Produk →
App & Web Consumer › Modul 01 › F03
Tap Shortcut Produk dari Grid Homepage
User melihat grid produk dan tap salah satu — misalnya "Transfer" atau "Pulsa". Flow ini dari tap di homepage sampai screen produk terbuka dan siap digunakan.
🎨
Referensi Figma
00 - Homepage — Section Product Grid 01 - Transfer E-Wallet Input 06 - Pulsa & Data Input
👤 User Login · Guest 🔔 User tap ikon produk di grid homepage ⏱ Pindah screen instan
1
Alur Normal
▾
1
User tap ikon produk di grid
Ada 8 ikon di grid. Setiap ikon membawa user ke screen yang berbeda:
  • Transfer → Screen 01 Transfer E-Wallet Input
  • Pulsa → Screen 06 Pulsa & Data Input
  • Token PLN → Screen 19 Token Listrik Input
  • Paket Data → Screen 06 Pulsa & Data (tab Paket Data)
  • BPJS → Pilihan screen: BPJS Kesehatan (08) atau Ketenagakerjaan (10)
  • Game → Screen 14 Voucher Game Pilih Game
  • E-Money → Screen 01 Transfer E-Wallet Input
  • Lainnya → Screen 28 Layanan Semua
2
Screen produk terbuka langsung
Navigasi ke screen baru terjadi instan — tidak ada loading di antara. Animasi slide dari kanan (standar navigasi mobile). Tombol kembali "← Beranda" ada di pojok kiri atas screen produk.
3
Data produk dimuat di screen tujuan
Begitu screen produk terbuka, data yang dibutuhkan baru dimuat (misalnya daftar nominal pulsa). Skeleton loading muncul sebentar di dalam screen produk, bukan saat transisi antar screen. Ini menjaga transisi tetap terasa cepat.
2
Aturan Bisnis
▾
📌
Semua produk di grid bisa diakses guest tanpa login. Tap produk tidak memicu prompt login. Login hanya diperlukan saat konfirmasi transaksi akhir — bukan di awal flow. Ini sesuai dengan prinsip guest-first GoPop.
📌
Produk "BPJS" di grid adalah shortcut ke submenu — bukan langsung ke screen input. User akan diminta pilih antara BPJS Kesehatan atau BPJS Ketenagakerjaan karena keduanya punya screen berbeda.
📌
Tap "Lainnya" membawa user ke screen 28 Layanan Semua yang menampilkan semua produk GoPop dalam tiga tab (Transfer, Semua, Tagihan). Dari sini user bisa lihat semua produk yang mungkin tidak ada di grid homepage.
3
Bad Cases
▾
Produk yang di-tap sedang nonaktif — dinonaktifkan Ipong dari dashboard karena gangguan supplier
Yang user lihat: Tap produk tapi tidak terjadi apa-apa, atau tampil pesan kesalahan.
Tampilkan bottom sheet kecil yang muncul dari bawah: "Layanan [nama produk] sedang tidak tersedia sementara. Kami sedang menangani gangguan. Coba lagi dalam beberapa saat." dengan tombol "Oke, Mengerti". Jangan tampilkan pesan teknikal seperti error code.
PIC: Agung + Fikly (implementasi pesan nonaktif yang ramah) · Ipong (kelola status produk dari dashboard)
User tap dengan cepat dua kali — screen produk terbuka dua kali (double navigation)
Yang user lihat: Dua screen produk terbuka bertumpuk — menekan "Kembali" dua kali untuk kembali ke homepage.
Nonaktifkan tombol tap selama animasi navigasi berlangsung (biasanya 300ms). Satu tap = satu navigasi. Ini standar yang perlu diimplementasi di semua tombol navigasi, bukan hanya grid produk ini.
PIC: Agung + Fikly (implementasi tap debounce / navigation guard)
4
Bottleneck
▾
Tap "BPJS" membawa user ke submenu pilihan — tapi tidak ada screen submenu BPJS di Figma saat ini
Saat ini Figma hanya punya screen 08 (BPJS Kesehatan Input) dan 10 (BPJS Ketenagakerjaan Input) — tidak ada screen "pilih dulu BPJS yang mana". Dua opsi: (1) Bili buat screen submenu BPJS baru di Figma, atau (2) langsung tampilkan bottom sheet pilihan saat user tap "BPJS" tanpa perlu screen baru. Opsi 2 lebih cepat diimplementasi.
PIC: Bili (tentukan solusi UX) · Agung + Fikly (implementasi)
5
Hasil Akhir
▾
✅ Flow Sukses
Screen produk terbuka instan dengan animasi slide
Data produk dimuat di background setelah screen terbuka
User siap melanjutkan ke flow transaksi produk tersebut
❌ Flow Gagal
Produk nonaktif → pesan ramah, bukan error teknikal
Double tap → dua screen terbuka tanpa navigation guard

F04: Tap Banner Promo →
App & Web Consumer › Modul 01 › F04
Tap Banner Promo di Homepage
Section "Promo Spesial 🔥" menampilkan banner carousel yang dikelola Kayong dari CMS. User bisa swipe untuk lihat banner lain atau tap banner untuk diarahkan ke tujuan yang relevan.
🎨
Referensi Figma
Banner terlihat di section "Promo Spesial 🔥" — horizontal carousel. Contoh di Figma: "Transfer DANA ke OVO - CASHBACK 5%" dan "BEBAS BIAYA ADMIN - Token PLN"
00 - Homepage (1:5452) — Section Promo Spesial
👤 User Login · Guest 🔔 User tap salah satu banner di carousel ⏱ Instan ke tujuan
1
Alur Normal
▾
1
Banner carousel tampil dan auto-rotate tiap 3 detik
Banner pertama langsung tampil saat homepage selesai load. Setiap 3 detik berganti ke banner berikutnya secara otomatis dengan animasi slide halus. Ada indikator titik kecil di bawah banner yang menunjukkan posisi saat ini dari total banner yang ada.
2
User swipe atau tap banner yang diinginkan
Swipe kiri/kanan untuk berganti banner secara manual. Saat user mulai swipe, auto-rotate berhenti sementara dan kembali lagi setelah 5 detik tidak ada interaksi. Tap banner untuk diarahkan ke tujuan yang sudah Kayong set saat upload.
3
User diarahkan ke tujuan sesuai link yang diset Kayong
Tiga jenis tujuan yang bisa diset di CMS:
  • Produk GoPop: Langsung buka screen produk (misalnya tap banner Transfer DANA ke OVO → buka screen Transfer E-Wallet)
  • Artikel: Buka halaman detail artikel Tips & Promo
  • Tidak ada link: Tap tidak menghasilkan aksi (banner hanya untuk branding)
2
Aturan Bisnis
▾
📌
Banner dikelola Kayong dari CMS dashboard. Kayong bisa upload gambar baru, set tujuan link, dan aktifkan/nonaktifkan banner kapan saja. Perubahan tampil di app user dalam maksimal 5 menit (waktu cache kadaluarsa).
📌
Maksimal 5 banner aktif sekaligus. Kalau Kayong tambah banner ke-6 tanpa nonaktifkan yang lama, sistem tolak dengan pesan "Nonaktifkan dulu salah satu banner yang ada."
📌
Konten banner tidak divalidasi sistem. Kayong bisa upload gambar dan teks apapun. Kayong dan tim GoPop bertanggung jawab memastikan konten sesuai dan sudah disetujui sebelum dipublish — terutama yang menyebutkan promo atau diskon spesifik.
3
Bad Cases
▾
Kayong set link banner ke produk yang sudah nonaktif — user tap banner tapi produk tidak bisa diakses
Yang user lihat: Tap banner, terbuka screen produk, tapi ada pesan "Layanan sedang tidak tersedia."
Ini perilaku yang benar — screen produk tetap terbuka tapi ada pesan nonaktif. Kayong perlu update atau nonaktifkan banner tersebut dari CMS segera setelah produk dinonaktifkan. Ke depan, sistem bisa auto-nonaktifkan banner yang mengarah ke produk nonaktif.
PIC: Kayong (update banner saat produk nonaktif) · Irwan + Rifky (fitur auto-nonaktif banner)
User tap banner promo cashback — tapi saat transaksi tidak dapat cashback yang dijanjikan
Yang user lihat: Total harga tidak ada diskon cashback meski banner bilang cashback 5%.
Ini masalah ekspektasi yang serius — bisa jadi keluhan ke CS. Penyebab paling umum: banner sudah kadaluarsa tapi belum dinonaktifkan, atau syarat cashback tidak terpenuhi (misalnya hanya untuk transaksi pertama). Kayong wajib selalu cantumkan syarat & ketentuan singkat di banner, dan nonaktifkan banner tepat waktu saat promo berakhir.
PIC: Kayong (disiplin nonaktifkan banner saat promo selesai) · Bili (SOP content approval sebelum publish)
4
Bottleneck
▾
Tidak ada sistem approval konten banner — Kayong bisa publish banner promo apapun tanpa perlu persetujuan dulu
Buat SOP sederhana: setiap banner yang menyebutkan angka (cashback %, diskon, atau "gratis") wajib dapat persetujuan Bili atau Acoy sebelum dipublish. Bisa lewat WA atau dicatat di sistem. Untuk banner yang hanya visual tanpa klaim khusus, Kayong bisa langsung publish.
PIC: Acoy (buat dan enforce SOP approval konten) · Kayong (patuhi SOP)
5
Hasil Akhir
▾
✅ Flow Sukses
Banner tampil dan auto-rotate tiap 3 detik
Swipe manual berfungsi dan menghentikan auto-rotate
Tap banner → navigasi ke screen produk atau artikel yang benar
❌ Flow Gagal
Banner mengarah ke produk nonaktif → pesan tidak tersedia
Promo banner tidak sesuai realita → user komplain ke CS

App & Web Consumer › Modul 01 › F05
Buka Halaman Layanan Semua
Screen 28 adalah direktori lengkap semua produk GoPop — dibagi tiga tab. User bisa ke sini dari navbar bawah (tap "Layanan") atau dari tap "Lainnya" di grid homepage.
🎨
Referensi Figma
28 — Tab Transfer 28b — Tab Semua 28c — Tab Tagihan
👤 User Login · Guest 🔔 Tap "Layanan" di navbar bawah atau tap "Lainnya" di grid
1
Alur Normal
▾
1
User tap "Layanan" di navbar bawah atau "Lainnya" di grid
Screen 28 terbuka. Default: tab "Transfer" aktif karena ini produk unggulan GoPop. Ikon "Layanan" di navbar bawah berubah menjadi aktif (warna ungu).
2
User melihat tiga tab: Transfer, Semua, Tagihan
Tab Transfer: Semua produk transfer e-wallet (GoPay, OVO, DANA, ShopeePay, dll)

Tab Semua: Semua produk GoPop dalam satu halaman — Transfer, Pulsa, Token PLN, Paket Data, BPJS, Voucher Game, Internet & TV, PDAM, PBB, dan lainnya

Tab Tagihan: Hanya produk postpaid/tagihan — BPJS, Internet & TV, PDAM, PBB, Listrik Pascabayar
3
User tap produk yang diinginkan
Langsung diarahkan ke screen input produk tersebut. Alur selanjutnya mengikuti flow transaksi masing-masing produk.
2
Aturan Bisnis
▾
📌
Tab default saat masuk adalah "Transfer" — bukan "Semua". Ini memprioritaskan produk utama GoPop yaitu Transfer E-Wallet flat Rp 1.000 yang menjadi differentiator utama dari kompetitor.
📌
Produk yang dinonaktifkan Ipong tidak muncul di halaman ini. User tidak akan melihat produk yang sedang gangguan — mengurangi frustrasi saat ada gangguan supplier.
📌
Halaman ini tidak memerlukan login. Guest bisa lihat semua produk dan tap untuk mulai transaksi.
3
Bad Cases
▾
Semua produk di tab "Tagihan" sedang nonaktif karena gangguan massal supplier — halaman tampil kosong
Yang user lihat: Tab "Tagihan" tampil kosong, tidak ada produk.
Tampilkan pesan informasi di area yang kosong: "Layanan tagihan sedang dalam perbaikan. Kami akan kembali segera. Cek layanan isi ulang kami sementara ini." dengan link ke tab "Semua" atau "Transfer". Jangan biarkan layar kosong tanpa penjelasan.
PIC: Agung + Fikly (empty state design) · Ipong + Kayong (koordinasi update banner homepage juga)
4
Bottleneck
▾
Produk baru yang ditambahkan Ipong di dashboard perlu muncul otomatis di halaman ini — saat ini tidak ada mekanisme sinkronisasi yang jelas
Daftar produk di halaman Layanan harus selalu sinkron dengan yang dikelola Ipong di dashboard. Kalau Ipong aktifkan produk baru, harus langsung muncul di sini tanpa perlu Agung/Fikly deploy ulang. Ini perlu disepakati antara Irwan/Rifky (BE) dan Agung/Fikly (FE) bagaimana cara sinkronisasinya.

App & Web Consumer › Modul 01 › F06
Homepage Saat Tidak Ada Koneksi Internet
User membuka app tapi tidak ada koneksi internet — atau koneksi putus tiba-tiba saat sedang di homepage. App harus tetap terbuka dan memberikan pengalaman yang baik, bukan error kosong.
👤 User Login · Guest 🔔 Buka app saat offline atau koneksi tiba-tiba putus
1
Alur Normal
▾
1
App deteksi tidak ada koneksi
Saat buka app atau koneksi putus di tengah jalan, app deteksi kondisi offline. Muncul banner tipis di bagian atas layar (di bawah status bar): "Kamu sedang offline" dengan warna oranye/kuning. Banner ini tidak memblokir konten di bawahnya.
2
Homepage tampil dari data yang tersimpan sebelumnya
Kalau user pernah buka app sebelumnya dan ada data tersimpan:
  • Grid 8 produk tampil dari cache (produk tidak berubah sering, aman untuk tampil dari cache)
  • Banner promo tampil dari cache — mungkin sudah lama, tapi lebih baik ada daripada kosong
  • Label kecil di bawah banner: "Konten mungkin tidak terkini"
Kalau belum pernah buka app sama sekali (pertama kali install, langsung offline): tampilkan homepage minimal dengan 8 produk hardcode saja, tanpa banner dan artikel.
3
User tap produk — muncul pesan tidak bisa lanjut
User masih bisa lihat homepage dan tap produk, tapi saat tap muncul bottom sheet: "Kamu sedang offline. Sambungkan ke internet untuk mulai transaksi." Tidak ada yang bisa dilakukan tanpa koneksi — transaksi butuh koneksi ke server.
4
Koneksi kembali — banner offline hilang, konten diperbarui
Saat koneksi pulih, banner "Kamu sedang offline" menghilang dengan animasi fade. App otomatis refresh konten di background tanpa user perlu tap apapun. Setelah konten baru datang, muncul snackbar kecil: "Koneksi kembali ✓" yang menghilang sendiri dalam 2 detik.
2
Aturan Bisnis
▾
📌
App tidak boleh tampil layar kosong atau error saat offline. Selalu ada konten yang bisa ditampilkan — entah dari cache atau dari data hardcode minimal. Pengalaman offline yang buruk (layar putih atau error teknikal) tidak dapat diterima.
📌
Tidak ada transaksi yang bisa dilakukan saat offline. Ini bukan limitasi teknikal yang perlu diatasi — memang harus begitu karena transaksi butuh verifikasi real-time ke server dan pembayaran ke payment gateway.
📌
Data tersimpan dari sesi sebelumnya bisa dipakai sampai 24 jam. Lebih dari itu, terlalu basi untuk ditampilkan (harga atau promo mungkin sudah berubah signifikan).
3
Bad Cases
▾
User sedang mengisi form transaksi (sudah input nomor HP dan nominal) lalu koneksi tiba-tiba putus
Yang user lihat: Form yang sudah diisi, banner offline muncul di atas.
Simpan data form yang sudah diisi di memori app (bukan dikirim ke server). Saat koneksi kembali, data masih ada di form — user bisa langsung lanjut tanpa isi ulang dari awal. Jangan hapus data form hanya karena offline. Tampilkan pesan: "Koneksi kembali. Data kamu masih tersimpan, lanjut transaksi sekarang?"
PIC: Agung + Fikly (simpan state form di memori saat offline)
App menampilkan harga dari cache yang sudah kadaluarsa — user kecewa harga berubah saat checkout
Yang user lihat: Harga pulsa Rp 6.500 dari cache, tapi saat online dan checkout harga berubah jadi Rp 7.000.
Ini bisa terjadi kalau Ipong update harga saat user offline. Saat koneksi kembali dan user masuk ke screen produk, selalu fetch harga terbaru dari server — jangan gunakan harga dari cache untuk screen produk. Harga di homepage memang boleh dari cache, tapi saat user masuk ke screen transaksi, wajib refresh harga.
PIC: Agung + Fikly (selalu fetch harga terbaru saat masuk screen produk)
4
Bottleneck
▾
Deteksi offline yang tidak akurat — app masih berpikir online padahal koneksi sangat lambat (bukan offline tapi sama buruknya)
Jangan hanya cek "ada/tidak ada koneksi" — cek juga apakah server bisa dijangkau dengan timeout pendek (2 detik). Koneksi yang sangat lambat (misalnya sinyal 2G) tetap terdeteksi sebagai "online" tapi API request akan timeout. Perlakukan keduanya sama: tampilkan data dari cache sambil terus coba refresh di background.
PIC: Agung + Fikly (implementasi network quality check, bukan hanya connectivity check)
5
Hasil Akhir
▾
✅ Flow Sukses
App tidak crash atau tampil layar kosong saat offline
User tahu sedang offline dari banner tipis di atas
Pesan ramah saat tap produk — bukan error teknikal
Saat online kembali, app langsung refresh tanpa restart manual
❌ Flow Gagal
App crash atau tampil layar putih kosong saat offline
Data form yang sudah diisi hilang saat koneksi putus
Harga dari cache berbeda dengan harga aktual saat online
