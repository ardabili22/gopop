Homepage App & Web › AW-F01
🏠 Load Homepage Pertama Kali
Flow ini terjadi setiap kali user atau guest membuka app GoPop atau mengunjungi gopop.id. Homepage perlu menampilkan: banner promo dari CMS, shortcut produk populer, dan section Tips & Promo — semuanya diambil dari server secara real-time.
👤 User / Guest 🔔 Trigger: Buka app atau kunjungi gopop.id ⏱ Target Load: <2 detik 🎨 Figma: 00 - Homepage
Alur Step by Step
1
App/web terbuka — tampilkan skeleton loading
Sambil menunggu data dari server, app langsung tampilkan skeleton (abu-abu berkedip) agar user tahu app sedang loading, bukan hang.
📱 Agung (App) / Fikly (Desktop)
Yang User Lihat
Layar muncul langsung dengan layout homepage tapi semua konten masih kosong/abu-abu berkedip (skeleton). Navbar bawah sudah muncul — user bisa tap menu lain sambil menunggu.
Yang Agung & Fikly Perlu Implement
Render skeleton component untuk: area banner (kotak abu tinggi), 6 skeleton shortcut produk (lingkaran + garis pendek), dan 2 skeleton card Tips & Promo. Jangan tampilkan halaman kosong — UX terasa lebih cepat dengan skeleton.
Status di Figma
Skeleton state belum ada di Figma. Ini perlu ditambahkan Bili. Sementara belum ada, Agung & Fikly bisa implement dengan warna abu #E5E7EB dengan animasi pulse sederhana.
2
FE kirim 2 request API secara bersamaan (paralel)
Supaya homepage lebih cepat, dua request dikirim sekaligus — tidak perlu menunggu yang satu selesai sebelum yang lain mulai.
📱 Agung / Fikly ⚙️ Irwan + Rifky (BE)
Yang Agung & Fikly Implement
Kirim dua request secara paralel menggunakan Promise.all() — jangan sequential (satu-satu). Dengan paralel, total waktu tunggu = waktu request terlama, bukan jumlah keduanya.
API Call #1 — Konten CMS Homepage
GET /cms/homepage
Ambil banner aktif dan artikel Tips & Promo yang sudah dipublish. Public endpoint — tidak butuh login. Dikerjakan Kayong/Luqman dari dashboard CMS.
Query Params
platform=app ← kirim "web" kalau dari browser desktop
Response
{ "banners": [ { "id": "BNR-001", "image_url": "https://cdn.gopop.id/banners/promo-hari-ini.jpg", "link_type": "product", // atau "url" atau "none" "link_value": "pulsa", // kategori produk yang dituju "active": true } ], "articles": [ { "id": "ART-001", "title": "5 Tips Hemat Pulsa Setiap Bulan", "thumbnail": "https://cdn.gopop.id/articles/tips-hemat.jpg", "slug": "tips-hemat-pulsa" } ] }
API Call #2 — Daftar Produk Populer
GET /products?popular=true&limit=6
Ambil 6 produk paling populer untuk ditampilkan sebagai shortcut di homepage. Public endpoint — tidak perlu login. Kalau user sudah login sebagai reseller, tambahkan token agar BE bisa return HPP sesuai tier.
Query Params
popular=true&limit=6 Authorization: Bearer {token} ← opsional, hanya kalau sudah login
Response
{ "products": [ { "id": "PROD-001", "name": "Transfer E-Wallet", "icon": "https://cdn.gopop.id/icons/ewallet.png", "category": "transfer", "is_hero": true, "tagline": "Flat Rp1.000" }, { "id": "PROD-002", "name": "Pulsa", "icon": "https://cdn.gopop.id/icons/pulsa.png", "category": "pulsa" } // ... 4 produk lainnya ] }
3
Response diterima — konten ditampilkan
Setelah data dari server datang, skeleton digantikan dengan konten nyata satu per satu sesuai data yang diterima.
📱 Agung / Fikly 👤 User
Yang User Lihat (Sesuai Figma Homepage)
  • Banner Promo — gambar dari CMS tampil sebagai carousel. Swipe kiri-kanan. Kalau tidak ada banner aktif dari CMS, tampil banner default hardcode.
  • Shortcut Transfer E-Wallet — tampil sebagai hero card paling atas dengan tagline "Flat Rp1.000" karena ini produk utama GoPop.
  • Grid 6 Produk Populer — ikon + nama produk dalam grid 3×2. Tap langsung ke flow transaksi produk tersebut.
  • Tips & Promo — 2–3 card artikel dari CMS. Tap ke halaman detail artikel.
Yang Agung & Fikly Perlu Implement
  • Banner: gunakan library carousel (misal Swiper.js). Auto-play setiap 3 detik. Indicator titik di bawah.
  • Cache response di localStorage dengan TTL 5 menit. Kalau user buka lagi dalam 5 menit, tampilkan cache dulu sambil fetch ulang di background.
  • Fallback: kalau image_url gagal load, tampilkan placeholder warna ungu dengan logo GoPop.
4
FE cek status login — tampilkan UI yang sesuai
Homepage tampilannya sedikit berbeda tergantung apakah user sudah login atau masih guest. Tidak ada perbedaan besar, tapi ada beberapa elemen yang muncul/hilang.
📱 Agung / Fikly
Perbedaan UI: Guest vs Login
Kalau Guest (tidak ada token di storage):
  • Header: "Selamat datang di GoPop 👋" + tombol "Masuk / Daftar"
  • Tidak ada greeting nama user
  • Semua produk tetap bisa di-tap (guest-first)
Kalau Sudah Login:
  • Header: "Halo, [Nama User] 👋"
  • Avatar kecil di pojok kanan atas
  • Tidak ada tombol "Masuk / Daftar"
Cek dari localStorage: ada access_token yang belum expired → sudah login.
Bottleneck & Cara Mengatasinya
⚡
4 Case yang Bisa Bermasalah
API GET /cms/homepage belum ada — homepage tidak bisa ambil konten CMS
Ini gap yang sudah teridentifikasi (INT-GAP-18 + AW-GAP-05). Sementara endpoint ini belum dibuat Irwan + Rifky, Agung & Fikly bisa hardcode data dummy di FE supaya bisa develop tampilan dulu. Tapi jangan ship ke production sebelum endpoint nyata tersedia.
⚙️ Fix: Irwan + Rifky buat endpoint GET /cms/homepage · 📱 Agung + Fikly: pakai mock data sementara untuk development
Server lambat merespons — user melihat homepage kosong terlalu lama
Pasang timeout 5 detik untuk kedua request. Kalau timeout: tampilkan konten dari cache localStorage (meski mungkin sudah lama). Kalau tidak ada cache sama sekali: tampilkan homepage versi minimal dengan hanya shortcut produk yang sudah di-hardcode (bukan dari API). User tetap bisa transaksi.
📱 Agung + Fikly: implementasi timeout + fallback ke cache + fallback ke hardcode minimal
Banner dari CMS menampilkan gambar yang tidak sesuai platform (resolusi salah)
Kayong upload 1 gambar tapi dipakai di app mobile (sempit) dan web desktop (lebar) — hasilnya bisa stretch atau crop tidak ideal. Solusinya: CMS support upload 2 versi gambar per banner: image_mobile dan image_desktop. FE app ambil image_mobile, FE web ambil image_desktop. Kalau hanya 1 yang diupload, gunakan yang ada untuk keduanya.
🎨 Bili (tambah spec gambar ke Figma) · 👤 Kayong (upload 2 versi di CMS) · Irwan+Rifky (update schema CMS) · Agung+Fikly (implementasi conditional)
Homepage version Figma belum dikunci — Agung dan Fikly tidak tahu mana yang diimplementasi
Ini blocker terbesar saat ini. Ada 13 versi "00 - Homepage" di Figma. Agung dan Fikly tidak bisa mulai implement sampai Bili pilih 1 dan rename jadi "00 - Homepage [FINAL]". Kalau ini tidak segera diselesaikan, Agung & Fikly tidak bisa mulai kerja pada flow ini sama sekali.
🎨 Bili: ACTION ITEM — pilih 1 versi final, rename, arsipkan sisanya. Estimasi 30 menit.
✅ Hasil Akhir Flow Ini
Homepage tampil dengan konten real dari server dalam <2 detik
Banner promo yang diubah Kayong dari CMS langsung muncul di app user
User/guest melihat UI yang sesuai status login mereka
Konten ter-cache untuk load lebih cepat saat buka lagi

Homepage App & Web › AW-F02
🔄 Refresh & Update Konten CMS
Kayong mengubah banner di dashboard CMS. Flow ini menjelaskan bagaimana perubahan itu akhirnya sampai ke layar user — termasuk bagaimana cache di-handle agar user tidak terus lihat konten lama.
📢 Kayong/Luqman → 👤 User 🔔 Trigger: Banner/konten diubah di CMS dashboard ⏱ Delay max: 5 menit (TTL cache)
Alur Step by Step
1
Kayong update banner di dashboard CMS
Kayong upload gambar baru atau nonaktifkan banner lama dari dashboard.
📢 Kayong (Marketing)⚙️ BE
Yang Terjadi di BE
Dashboard kirim request update ke BE → BE simpan perubahan di database → BE invalidate cache untuk key cms:homepage:app dan cms:homepage:web di Redis.
API Call dari Dashboard CMS
PUT /admin/cms/banners/{banner_id}
Update atau nonaktifkan banner. Butuh login admin dengan permission cms.write.
Request Body
{ "image_url": "https://cdn.gopop.id/banners/promo-baru.jpg", "active": true, "platform": "app", "link_type": "product", "link_value": "transfer" }
Response
{ "success": true, "banner_id": "BNR-002", "cache_invalidated": true // BE sudah hapus cache lama }
2
User buka / refresh homepage
Saat user buka homepage, FE cek dulu apakah ada cache yang masih valid sebelum call API.
📱 Agung / Fikly
Cache Strategy yang Harus Diimplementasi
Langkah yang FE lakukan setiap buka homepage:
  • Cek localStorage: ada data homepage_cache yang timestamp-nya kurang dari 5 menit lalu?
  • Kalau ya (cache masih fresh): tampilkan data cache dulu → di background, tetap fetch ulang dari server → kalau data baru berbeda, update tampilan tanpa user perlu refresh manual
  • Kalau tidak (cache expired atau tidak ada): tampilkan skeleton → fetch dari server → tampilkan data baru → simpan ke cache dengan timestamp baru
Pattern ini disebut stale-while-revalidate — user tidak pernah nunggu lama tapi konten selalu up-to-date.
API Call dari FE (sama dengan AW-F01)
GET /cms/homepage?platform=app
BE cek Redis cache dulu. Cache key: cms:homepage:app. Cache TTL di server: 5 menit. Kalau cache sudah di-invalidate Kayong, BE ambil fresh dari database dan set cache baru.
3
Banner baru muncul di layar user
User melihat banner baru tanpa perlu tahu bahwa ada perubahan — update terasa mulus.
👤 User
Yang User Lihat
Kalau user sudah ada di homepage saat update terjadi — banner berganti pelan-pelan (fade transition) tanpa halaman reload. Kalau user baru buka homepage — langsung lihat banner baru.
Bottleneck & Cara Mengatasinya
⚡
2 Case yang Bisa Bermasalah
Kayong ubah banner tapi user masih lihat yang lama sampai 5 menit ke depan
Ini wajar dan by design — TTL cache 5 menit. Kalau Kayong perlu update yang langsung live untuk kampanye urgent, ada tombol "Force Refresh Cache" di dashboard CMS yang akan invalidate cache segera. Untuk update normal, 5 menit sudah cukup.
📢 Kayong: pakai tombol "Force Refresh" kalau butuh langsung live · ⚙️ Irwan+Rifky: implementasi endpoint invalidate cache on-demand
CMS dashboard belum terhubung ke BE — banner yang diubah tidak pernah sampai ke user
Ini adalah gap kritis (D-GAP-18). Saat ini semua perubahan CMS hanya update state lokal di browser Kayong — tidak tersimpan ke database, tidak ada yang bisa diambil FE. Kayong bisa ubah banner sebanyak apapun tapi user tidak akan pernah lihat perubahannya sampai ini difix.
⚙️ Fix: Irwan+Rifky buat semua endpoint CMS dan hubungkan ke dashboard · Ini harus selesai sebelum fitur CMS bisa dipakai Kayong
✅ Hasil Akhir Flow Ini
Perubahan banner Kayong di CMS muncul di app user dalam max 5 menit
User tidak perlu refresh manual — update otomatis
Performance tetap cepat karena ada caching di FE dan BE

Homepage App & Web › AW-F03
👆 User Tap Shortcut Produk dari Homepage
User melihat grid produk di homepage dan tap salah satu — misalnya "Transfer E-Wallet" atau "Pulsa". Flow ini menjelaskan apa yang terjadi dari tap sampai screen produk muncul.
👤 User / Guest 🔔 Trigger: User tap ikon produk di homepage ⏱ Target: Langsung pindah screen tanpa loading
Alur Step by Step
1
User tap ikon produk
Setiap ikon produk di homepage punya category dan link_type yang menentukan ke mana user diarahkan.
👤 User / Guest📱 Agung / Fikly
Routing Logic di FE (Berdasarkan category dari API)
  • category: "transfer" → Screen 01 Transfer E-Wallet Input
  • category: "pulsa" → Screen 06 Pulsa & Data Input
  • category: "token_listrik" → Screen 19 Token Listrik Input
  • category: "game" → Screen 14 Voucher Game Pilih Game
  • category: "bpjs" → Screen 08 BPJS Kesehatan Input
  • category: "pdam" → Screen 17 PDAM Input
  • category: "all" → Screen 28 Layanan Semua
Navigasi ini langsung — tidak perlu API call dulu. Screen langsung buka.
2
Screen produk terbuka — fetch data produk
Setelah screen produk terbuka, baru FE fetch data produk yang relevan (misal: daftar nominal pulsa). Ini dipisah dari homepage agar homepage tetap ringan.
📱 Agung / Fikly⚙️ BE
Contoh: User Tap "Pulsa"
Screen 06 Pulsa Input terbuka. Ada field input nomor HP. Saat user mulai ketik nomor → FE hit API untuk detect operator dari prefix nomor.
API: Detect Operator dari Nomor HP (Real-time)
GET /products/prefix-detect?phone=0812
Dipanggil saat user ketik 4 digit pertama nomor HP. Tidak perlu nomor lengkap. Public endpoint, tidak butuh login.
Response
{ "operator": "Telkomsel", "operator_logo": "https://cdn.gopop.id/operators/telkomsel.png", "prefix": "0812", "is_valid": true } // Kalau prefix tidak dikenal: { "operator": null, "is_valid": false, "message": "Prefix nomor tidak dikenal" }
API: Daftar Nominal Pulsa (Setelah Operator Terdeteksi)
GET /products?category=pulsa&operator=telkomsel
Ambil semua nominal pulsa untuk operator yang terdeteksi. Public — tidak butuh login. Kalau sudah login sebagai reseller, BE return HPP sesuai tier.
Response
{ "products": [ { "id": "PLN-TL-5", "name": "Pulsa Telkomsel 5.000", "nominal": 5000, "price": 6500, // harga jual consumer "hpp": null // null kalau bukan reseller, atau angka kalau reseller }, { "id": "PLN-TL-10", "name": "Pulsa Telkomsel 10.000", "nominal": 10000, "price": 11500 } // ... nominal lainnya ] }
Bottleneck & Cara Mengatasinya
⚡
2 Case yang Bisa Bermasalah
Produk di-tap dari banner CMS yang link-nya salah atau produk sudah tidak aktif
Saat FE terima response GET /products?category=... dan produk tidak ada atau active: false — tampilkan pesan: "Layanan ini sedang tidak tersedia. Coba lagi nanti." dengan tombol kembali ke homepage. Jangan tampilkan halaman kosong atau error teknikal.
📱 Agung + Fikly: implementasi empty state yang friendly · Irwan+Rifky: pastikan response selalu ada field active
Prefix detect terlalu sering di-hit saat user ketik — bikin boros request
Jangan hit API setiap karakter diketik. Pakai debounce 500ms — tunggu sampai user berhenti mengetik 0.5 detik baru kirim request. Dan hanya hit kalau sudah 4 digit (prefix selalu 4 digit). Ini mengurangi 80% request yang tidak perlu.
📱 Agung + Fikly: implementasi debounce di input field nomor HP
✅ Hasil Akhir Flow Ini
User langsung masuk ke screen produk yang ditap tanpa loading awal
Operator otomatis terdeteksi saat user ketik 4 digit pertama nomor HP
Daftar nominal muncul sesuai operator — siap dipilih user

Homepage App & Web › AW-F04
📶 Homepage Saat Tidak Ada Koneksi
User buka app tapi tidak ada koneksi internet. GoPop tetap harus bisa dibuka — user tidak boleh lihat layar error kosong. Flow ini menjelaskan bagaimana app handle kondisi offline dengan graceful.
👤 User / Guest 🔔 Trigger: App dibuka saat offline / koneksi putus ⏱ Target: App tetap terbuka, informasikan status offline
Alur Step by Step
1
FE deteksi tidak ada koneksi
Saat app dibuka atau koneksi tiba-tiba putus, FE harus deteksi kondisi ini dan bereaksi yang tepat.
📱 Agung / Fikly
Cara Deteksi Offline
  • Listen ke event window.addEventListener('offline', ...) — trigger saat koneksi putus di tengah jalan
  • Saat app buka, coba fetch dengan timeout pendek (2 detik) — kalau gagal, anggap offline
  • Tampilkan banner tipis di bagian atas: "Kamu sedang offline. Beberapa fitur mungkin tidak tersedia."
2
Tampilkan homepage dari cache
Kalau ada data homepage yang tersimpan dari sesi sebelumnya, tampilkan itu. User tetap bisa lihat UI penuh.
📱 Agung / Fikly
Yang Ditampilkan Saat Offline
Kalau ada cache (paling ideal):
  • Banner promo dari cache — mungkin sudah lama, tapi lebih baik dari kosong
  • Grid produk dari cache — produk tidak berubah sering jadi aman
  • Label kecil di bawah banner: "Konten mungkin tidak terkini"
Kalau tidak ada cache sama sekali (pertama kali install, belum pernah online):
  • Tampilkan homepage minimal: hanya grid produk yang di-hardcode (tidak dari API)
  • Banner tidak tampil — bukan placeholder, cukup tidak ada
Yang User Lihat & Bisa Lakukan
User masih bisa lihat homepage, tapi kalau tap produk → muncul pesan: "Tidak bisa melanjutkan — kamu sedang offline. Sambungkan ke internet untuk transaksi." Ini mencegah user frustrasi karena tidak tahu kenapa tidak bisa lanjut.
3
Koneksi kembali — app otomatis update
Saat koneksi pulih, app otomatis refresh konten tanpa perlu user tap apapun.
📱 Agung / Fikly
Implementasi
Listen ke event window.addEventListener('online', ...) → saat online kembali, auto-fetch data terbaru → update tampilan → hilangkan banner "offline" → tampilkan snackbar kecil: "Koneksi kembali ✓"
Bottleneck & Cara Mengatasinya
⚡
1 Case yang Perlu Diantisipasi
User tap produk saat offline lalu koneksi kembali — state produk hilang
Simpan state "user terakhir tap produk X" di memory. Saat koneksi kembali, jangan otomatis lanjut ke transaksi (bisa expired atau state tidak konsisten). Cukup biarkan user di screen produk yang sama — mereka bisa tap lagi untuk mulai. Tampilkan snackbar: "Koneksi kembali. Kamu bisa lanjut transaksi sekarang."
📱 Agung + Fikly: jangan auto-resume transaksi yang terputus — biarkan user yang inisiasi ulang
✅ Hasil Akhir Flow Ini
App tidak crash atau tampil kosong saat offline
User tahu kenapa tidak bisa transaksi (pesan offline yang jelas)
Saat online kembali, app langsung siap tanpa perlu restart
