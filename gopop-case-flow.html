API endpoint, logic server, error codes, database operation, queue/scheduler yang diperlukan.

A1
Register Consumer
End User · gopop.id & App · Guest-first, registrasi opsional
Happy Path Registrasi berhasil — akun aktif via OTP WhatsApp ▾
Bisnis
User baru daftar akun untuk simpan riwayat & kumpulkan poin loyalty. Registrasi tidak wajib untuk transaksi — ini soft acquisition. Satu nomor HP = satu akun consumer.
UX
  • 1
    Tap "Daftar"
    Dari soft prompt post-transaksi atau dari halaman Profil
  • 2
    Isi form
    Nama lengkap · Nomor HP · Password (min 8 karakter, huruf+angka)
  • 3
    OTP dikirim via WhatsApp
    6 digit · berlaku 5 menit · ada countdown timer di layar
  • 4
    Input OTP → Akun aktif
    Redirect ke Home, sudah dalam status login
Backend
POST /auth/register → cek duplikasi nomor HP → simpan user status UNVERIFIED → trigger WA OTP via WhatsApp Business API

POST /auth/verify-otp → validasi OTP + TTL → update status ACTIVE → return access_token (15 menit) + refresh_token (30 hari)
E1 Nomor HP sudah terdaftar ▾
Bisnis
Satu nomor HP hanya boleh terdaftar satu kali sebagai consumer. Duplikasi dilarang.
UX
Setelah submit form → pesan spesifik: "Nomor ini sudah terdaftar. Masuk sekarang?" dengan tombol langsung ke halaman Login. Bukan pesan generic "terjadi kesalahan."
Backend
POST /auth/register return 400 PHONE_ALREADY_EXISTS → frontend handle dengan copy yang spesifik. Jangan expose apakah nomor exist ke request yang tidak terautentikasi secara berlebihan — cukup di form register.
E2 OTP tidak datang atau expired ▾
Bisnis
OTP valid 5 menit. Resend maksimal 3x per sesi dengan cooldown 60 detik antar resend. Setelah 3x → cooldown 10 menit.
UX
Countdown timer tampil real-time. Setelah 5 menit → tombol "Kirim Ulang" aktif. Setelah 3x resend → "Coba lagi dalam 10 menit" + opsi "Hubungi CS".
Backend
OTP disimpan di Redis dengan TTL 5 menit. Rate limit resend per nomor HP di Redis: otp:resend:{phone} → counter + TTL 10 menit. Setelah 3x → return 429 OTP_RESEND_LIMIT dengan retry_after timestamp.
E3 OTP salah berkali-kali ▾
Bisnis
Maksimal 5x percobaan input OTP. Lewat batas → sesi hangus, user harus mulai ulang dari awal.
UX
Setiap salah → "Kode salah, sisa X percobaan." Setelah 5x salah → "Sesi habis, mohon daftar ulang" → redirect ke form register (field bersih).
Backend
Counter attempt per OTP session di Redis: otp:attempt:{phone}. Setelah 5x → PUT /auth/otp → 429 OTP_MAX_ATTEMPT → invalidate OTP + hapus session. User harus register ulang.
A2
Login Consumer
End User yang sudah punya akun · Akses riwayat, poin, profil
🎨 Bili (UI/UX)⚙️ Irwan + Rifky (BE)⚙️ Agung (App)⚙️ Fikly (Desktop)👁️ Acoy (Monitor)
Happy Path Login berhasil ▾
UX
Input nomor HP + password → jika sukses → redirect ke halaman sebelumnya (atau Home). Sesi tersimpan otomatis.
Backend
POST /auth/login → validasi credentials → return access_token (15 menit) + refresh_token (30 hari). Frontend simpan token di secure storage. Token menyertakan role: "consumer".
E1 Password salah — risiko brute force ▾
Bisnis
Maksimal 5x salah dalam 15 menit → akun terkunci sementara 30 menit. Untuk mencegah brute force attack.
UX
Tiap salah → "Kata sandi salah, sisa X percobaan." Setelah lock → "Akun dikunci sementara. Coba lagi dalam 30 menit atau reset kata sandi." Ada tombol "Lupa Kata Sandi" yang selalu visible.
Backend
Login attempt counter di Redis: login:attempt:{phone} TTL 15 menit. Setelah 5x → set account_locked_until di DB (timestamp + 30 menit). POST /auth/login return 423 ACCOUNT_LOCKED dengan field retry_after.
Dashboard
CS bisa lihat status LOCKED di profil pengguna dan manually unlock dari menu Pengguna jika user komplain sebelum 30 menit habis.
E2 Akun belum verifikasi OTP ▾
Bisnis
Akun status UNVERIFIED tidak boleh bisa transaksi. Harus verifikasi nomor HP dulu.
UX
Login credentials benar tapi muncul screen/banner: "Akun belum diverifikasi. Kirim ulang kode ke nomor HP kamu?" dengan tombol resend OTP. Tidak langsung masuk app.
Backend
POST /auth/login return 200 tapi dengan flag verified: false → frontend intercept dan redirect ke OTP verification screen. Token tidak diberikan sampai verifikasi selesai.
E3 Akun di-suspend admin ▾
Bisnis
Admin bisa suspend akun consumer dari dashboard karena fraud, abuse, atau pelanggaran lain.
UX
"Akun kamu sementara dinonaktifkan. Hubungi CS GoPop untuk informasi lebih lanjut." Ada tombol langsung ke WA CS. Tidak ada detail alasan suspend yang ditampilkan ke user.
Backend
POST /auth/login return 403 ACCOUNT_SUSPENDED → frontend tampil pesan + CTA WA CS.
Dashboard
CS lihat status SUSPENDED di profil pengguna, bisa lihat catatan alasan suspend (internal note), dan bisa reaktivasi akun jika sudah clear.
E4 Session expired di tengah penggunaan app ▾
Bisnis
Access token expired (15 menit) → harus pakai refresh token. Kalau refresh token juga expired (30 hari) → harus login ulang. Jangan paksa user logout tiba-tiba tanpa konteks.
UX
Refresh token harus jalan silent di background — user tidak boleh sadar ada proses ini. Kalau refresh gagal → simpan state (halaman mana, sudah input apa) → redirect login dengan pesan: "Sesi kamu habis, silakan masuk lagi. Progress kamu tersimpan."
Backend
HTTP interceptor di frontend: setiap API call return 401 → otomatis hit POST /auth/refresh dengan refresh token → kalau berhasil, ulangi request original dengan token baru. Kalau refresh return 401 juga → logout paksa, clear semua token, redirect login.
A3
Register Reseller
Reseller App · Perlu approval admin · Default tier Bronze
🎨 Bili (UI/UX)⚙️ Irwan + Rifky (BE)⚙️ Dileh (Reseller App)⚙️ Isal + Ipong (Dashboard)👁️ Acoy (Monitor)
Happy Path Pendaftaran reseller berhasil dan diapprove admin ▾
Bisnis
Reseller adalah akun B2B dengan fitur khusus: saldo prepaid, HPP, downline. Perlu approval manual admin sebelum bisa transaksi. Tier default Bronze. Wallet dibuat saat approved dengan saldo Rp 0.
UX
  • 1
    Isi form pendaftaran
    Nama, nomor HP, nama usaha, alamat, kode referral (opsional)
  • 2
    Submit → Screen waiting
    "Pendaftaran kamu sedang diproses. Tim kami akan menghubungi kamu dalam 1×24 jam."
  • 3
    WA notif saat approved
    "Akun reseller kamu aktif! Silakan login dan mulai top up saldo."
  • 4
    Login → Dashboard reseller
    Saldo Rp 0 · Tier Bronze · Referral code sudah ter-generate
Backend
POST /resellers/register → simpan status PENDING_APPROVAL → notif badge di dashboard admin (menu Reseller).

PUT /admin/resellers/{id}/approve → update status ACTIVE → buat wallet record (balance: 0, total_omzet: 0) → assign tier Bronze → generate referral_code unik 8 karakter → kirim WA notif ke reseller.
Dashboard
Menu Reseller → tab "Pending Approval" → list semua pendaftar baru. Admin bisa lihat detail, approve, atau reject dengan catatan alasan.
E1 Nomor HP sudah terdaftar sebagai consumer ▾
Bisnis
⚠️ Gap keputusan (A-G1): Apakah 1 nomor HP bisa jadi consumer sekaligus reseller? Perlu keputusan Bili. Handling sementara di bawah.
UX (sementara)
"Nomor ini sudah terdaftar sebagai pengguna GoPop. Hubungi CS untuk upgrade ke akun reseller." Ada tombol WA CS langsung.
Backend (sementara)
Return 400 PHONE_EXISTS_AS_CONSUMER sampai ada keputusan bisnis apakah merge account atau tetap block.
E2 Pendaftaran ditolak admin ▾
Bisnis
Admin bisa tolak pendaftaran: data tidak lengkap, wilayah duplikat, atau alasan lain. CS follow-up via WA.
UX
WA notif: "Pendaftaran reseller kamu belum bisa diproses saat ini. Hubungi CS GoPop untuk informasi lebih lanjut." Tidak ada alasan detail di WA — CS yang jelaskan.
Backend
PUT /admin/resellers/{id}/reject → status REJECTED → trigger WA notif. Alasan penolakan disimpan sebagai internal note di DB — tidak dikirim ke user.
Dashboard
Admin wajib isi alasan sebelum reject (dropdown + notes). Alasan tersimpan di profil reseller untuk referensi CS saat follow-up.
E3 Login saat status masih PENDING ▾
Bisnis
Reseller pending tidak boleh akses fitur transaksi apapun.
UX
Masuk app → full-screen state: "Pendaftaran kamu masih diproses" dengan estimasi waktu dan kontak CS. Tidak ada menu navigasi aktif.
Backend
GET /resellers/profile return status: "PENDING_APPROVAL" → frontend render pending state screen. Semua endpoint transaksi return 403 ACCOUNT_PENDING.
A4
Login Reseller + PIN App
Reseller App · PIN sebagai second layer security
🎨 Bili (UI/UX)⚙️ Irwan + Rifky (BE)⚙️ Dileh (Reseller App)
Happy Path Login + PIN berhasil ▾
Bisnis
Reseller App punya PIN 6 digit sebagai layer keamanan kedua (seperti e-wallet). PIN diminta setiap buka app dan setiap konfirmasi transaksi.
UX
Login password seperti biasa → berhasil → redirect ke PIN screen. Buka app setelah background → langsung ke PIN screen (tidak perlu password lagi selama session aktif).
Backend
PIN disimpan hashed di backend — bukan di device. Validasi via POST /resellers/auth/verify-pin. Return short-lived session token untuk authorize transaksi.
E1 PIN salah 3x — dikunci ▾
Bisnis
Salah PIN 3x → PIN dikunci. Harus re-auth dengan password untuk bisa set PIN baru. Ini cegah akses tidak sah jika HP hilang.
UX
Setiap salah → counter tampil. Setelah 3x → "PIN dikunci. Gunakan kata sandi untuk masuk dan atur ulang PIN." → redirect ke login password.
Backend
POST /resellers/auth/verify-pin → counter di Redis. Setelah 3x → return 423 PIN_LOCKED → user harus POST /auth/login dengan password → set PIN baru via PUT /resellers/auth/pin.

Consumer & Reseller · Reset via OTP WhatsApp
Happy Path Reset password berhasil ▾
UX
  • 1
    Tap "Lupa Kata Sandi"
    Di halaman login
  • 2
    Input nomor HP
  • 3
    OTP dikirim via WA
    6 digit · 5 menit
  • 4
    Input OTP → Input password baru (2x konfirmasi)
  • 5
    Berhasil → auto login
Backend
POST /auth/forgot-password → cek nomor terdaftar → kirim OTP.

POST /auth/reset-password → validasi OTP + nomor HP → update password hash → invalidate semua session aktif (force logout semua device) → return token baru untuk auto-login.
E1 Nomor HP tidak terdaftar ▾
Bisnis
Jangan expose apakah nomor terdaftar atau tidak ke publik — ini cegah user enumeration attack.
UX
"Kalau nomor ini terdaftar, kamu akan menerima kode via WhatsApp." Response yang sama baik nomor ada maupun tidak.
Backend
Return 200 regardless nomor ada atau tidak. Kalau tidak ada → tidak kirim OTP tapi response tetap sama. Tambahkan artificial delay yang konsisten untuk cegah timing attack.
T1
Transaksi Consumer — Happy Path
Semua produk prepaid & postpaid · Web & App consumer
Happy Path Produk Prepaid (Pulsa, Token PLN, Voucher Game) — Sukses ▾
Bisnis
Consumer bayar harga jual (HPP + margin GoPop). Payment gateway handle pembayaran dulu, baru backend kirim ke biller. GoPop ambil margin dari selisih. Status lifecycle: PENDING → PAID → PROCESSING → SUCCESS.
UX
  • 1
    Pilih kategori & produk
    Harga tampil dari catalog API real-time
  • 2
    Input nomor tujuan
    Nomor HP / ID Pelanggan / Nomor meter. Validasi format di frontend.
  • 3
    Konfirmasi detail + harga final
    Preview produk, nominal, harga total. User harus confirm sebelum lanjut.
  • 4
    Pilih metode bayar
    QRIS · VA · E-Wallet (GoPay/OVO/DANA) · Retail (Alfamart/Indomaret)
  • 5
    Payment Instruction screen
    Tampil kode bayar + countdown timer. App polling status tiap 5 detik.
  • 6
    User bayar di luar app
  • 7
    Polling detect PAID → "Sedang diproses..."
  • 8
    Receipt screen
    Detail transaksi · Nomor referensi · Tombol "Simpan Struk" · WA notif otomatis
  • 9
    Soft prompt daftar akun (untuk guest)
Backend
POST /transactions → validasi produk aktif + harga masih valid → buat transaksi PENDING → POST /payments → hit Xendit → return payment instruction.

Xendit callback → POST /payments/callback → update PAID → queue job kirim ke biller (Ayoconnect primary) → response sukses → update SUCCESS → trigger WA notif.

Frontend polling: GET /transactions/{id} setiap 5 detik selama di Payment Instruction screen. Stop polling setelah status SUCCESS atau FAILED.
Dashboard
Transaksi muncul real-time di Monitoring dengan status lifecycle lengkap. Field source_channel: "web" atau "app" ter-tag otomatis untuk keperluan rekonsiliasi Rifky.
Happy Path Produk Postpaid / Tagihan (PLN Pascabayar, BPJS, PDAM) — Sukses ▾
Bisnis
Tagihan butuh satu step tambahan: inquiry (cek tagihan dulu sebelum bayar). Inquiry bukan transaksi — tidak ada biaya, tidak ada record di DB transaksi.
UX
Input ID Pelanggan → tap "Cek Tagihan" → loading inquiry → tampil: nama pelanggan, periode tagihan, jumlah tagihan, denda (jika ada) → user konfirmasi → lanjut ke pilih metode bayar (flow sama seperti prepaid).
Backend
POST /transactions/inquiry → hit Ayoconnect inquiry API → return nama + nominal tagihan. Inquiry tidak buat record transaksi.

Setelah konfirmasi user → POST /transactions dengan data dari hasil inquiry. Flow bayar sama seperti prepaid.
T2
Payment Gateway Gagal
Sebelum uang masuk ke GoPop · Tidak ada refund yang diperlukan
🎨 Bili (UI/UX)⚙️ Irwan + Rifky (BE)⚙️ Agung (App)⚙️ Fikly (Desktop)
T2-A QRIS Expired — 15 menit tidak dibayar ▾
Bisnis
QRIS dari Xendit punya TTL 15 menit. Kalau expired, transaksi dibatalkan. Tidak ada uang yang masuk ke GoPop — tidak perlu refund. Status: EXPIRED bukan FAILED.
UX
Countdown timer habis → "Waktu pembayaran habis. Transaksi dibatalkan."
Tombol: "Coba Lagi" (balik ke pilih metode bayar, data prefilled) + "Kembali ke Beranda". Tidak ada WA notif karena tidak ada uang masuk.
Backend
Frontend stop polling saat countdown habis → PUT /transactions/{id} update status EXPIRED. Xendit otomatis expire QRIS di sisi mereka.

Record tetap ada di DB untuk audit tapi tidak muncul di riwayat user (atau muncul dengan label "Kedaluwarsa" yang berbeda dari "Gagal").
T2-B VA Expired — 24 jam tidak dibayar ▾
Bisnis
VA berlaku 24 jam. Handling identik dengan T2-A — tidak ada uang masuk, tidak ada refund.
UX
Untuk VA, user mungkin tidak aktif di app saat expired. Bisa ada push notif 1 jam sebelum expired sebagai reminder: "VA kamu untuk [produk] hampir kedaluwarsa dalam 1 jam."
Backend
Scheduler: cek VA yang akan expired dalam 1 jam → kirim push notif reminder. Saat Xendit callback expired → update status EXPIRED.
T2-C E-Wallet Redirect — User batalkan atau tidak selesaikan di app e-wallet ▾
Bisnis
User di-redirect ke GoPay/OVO/DANA tapi tidak selesaikan pembayaran (tutup app, batalkan, saldo tidak cukup). Tidak ada uang masuk.
UX
User balik ke GoPop app → cek status → belum terbayar → "Pembayaran belum selesai. Selesaikan via [nama e-wallet] atau pilih metode lain."
Tombol: "Coba Bayar Lagi" (re-trigger redirect) + "Ganti Metode Bayar" + "Batalkan".
Backend
Saat user balik ke app → GET /payments/{id} → status masih PENDING.
Ganti metode: PUT /transactions/{id}/change-payment → buat payment baru, batalkan payment lama di Xendit.
Batalkan: PUT /transactions/{id} → status CANCELLED.
T2-D Webhook Xendit tidak datang — User sudah bayar tapi status tidak update ▾
Bisnis
User sudah bayar di bank/e-wallet tapi GoPop tidak dapat notifikasi dari Xendit karena network issue atau Xendit timeout. Ini jarang tapi bisa terjadi dan sangat meresahkan user.
UX
Setelah 10 menit tanpa status update → tampil: "Pembayaran kamu sedang diverifikasi. Kami akan kirim konfirmasi via WhatsApp. Nomor referensi: [ID]". Jangan biarkan user terlalu lama di layar loading tanpa info.
Backend
Scheduler tiap 5 menit: cek transaksi PENDING yang sudah >10 menit → hit Xendit API aktif untuk cek status payment (pull, bukan tunggu push).
Xendit confirm paid → proses normal (update PAID → kirim ke biller).
Xendit bilang belum bayar → biarkan sampai expire.
Dashboard
Rifky/CS bisa lihat transaksi PENDING >10 menit dengan flag khusus di tab "Perlu Perhatian". Ada tombol "Cek Status Manual ke Xendit" untuk trigger pull manual.

⚠️ Paling kritis — uang sudah di GoPop tapi produk belum terkirim
⚙️ Irwan + Rifky (BE)⚙️ Isal + Ipong (Dashboard)👁️ Acoy (Monitor)
T3-A Biller Primary (Ayoconnect) Timeout — Auto retry + fallback ▾
Bisnis
GoPop sudah terima uang dari user. Ayoconnect tidak response dalam batas waktu. GoPop harus: (1) retry ke Ayoconnect, (2) fallback ke biller backup, (3) kalau semua gagal → refund user. Reputasi GoPop ada di flow ini.
UX
User tidak boleh tahu detail teknis (biller timeout, fallback, dll). Yang user lihat: "Transaksi kamu sedang diproses. Kami akan konfirmasi via WhatsApp dalam beberapa menit."

Jangan tampil "Gagal" sebelum semua retry dan fallback sudah dicoba. Kalau akhirnya gagal total → "Transaksi tidak dapat diselesaikan. Dana kamu akan dikembalikan dalam 1×24 jam."
Backend
Setelah PAID → hit Ayoconnect timeout 30 detik → timeout → retry 2x (exponential backoff: 30 detik, 60 detik) → masih timeout → fallback ke Digiflazz/IAK (biller backup).

Backup sukses → SUCCESS, user tidak tahu ada fallback.
Semua gagal → status FAILED → trigger auto-refund flow.

Field retry_count di-increment setiap attempt. Catat di log: biller mana yang dicoba, hasil apa, waktu berapa.
Dashboard
Transaksi tampil status PROCESSING selama retry. Setelah fallback ada log: "Switched to backup biller: Digiflazz". Kalau stuck >15 menit → auto-flag ke tab "Perlu Perhatian" + notif ke Ops (Acoy).
T3-B Biller: Nomor tidak valid / tidak ditemukan ▾
Bisnis
User input nomor yang salah (tidak aktif, ID pelanggan tidak exist). Biller tolak request. Produk tidak dikirim. Uang sudah masuk ke GoPop → harus refund penuh.
UX
