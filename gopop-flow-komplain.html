Komplain & Tiket › D-F01
📝 User Submit Tiket dari Pusat Bantuan
Flow ini dimulai ketika user atau guest punya masalah dan ingin melaporkan ke GoPop. Mereka buka Pusat Bantuan di app atau web, isi form, lalu kirim. Tidak perlu login — guest pun bisa komplain.
👤 User / Guest 🔔 Trigger: User punya masalah ⏱ Estimasi: 2–3 menit
Alur Step by Step
1
User buka menu Pusat Bantuan
User tap tombol "Pusat Bantuan" dari menu navigasi bawah, atau tap langsung dari screen transaksi gagal (ada shortcut di sana).
👤 User
Yang User Lihat & Lakukan
Halaman Pusat Bantuan terbuka. Ada form dengan 3 bagian yang harus diisi. Di bagian atas ada info singkat: "Kami siap bantu. Tim GoPop akan menghubungi kamu via WhatsApp."
2
User isi form komplain
Ada 3 field yang harus diisi. Field nomor WA adalah satu-satunya cara CS menghubungi user, jadi wajib diisi bahkan untuk user yang sudah login.
👤 User / Guest
Yang User Lihat & Lakukan
  • Nomor WhatsApp Aktif (wajib) — nomor yang bisa dihubungi CS. Kalau sudah login, otomatis terisi dari profil tapi bisa diganti.
  • Kategori Masalah (wajib) — dropdown pilihan: Transaksi Gagal, Saldo Tidak Masuk, Masalah Akun/Login, Lainnya.
  • ID Transaksi (opsional) — bisa dikosongkan kalau tidak ingat atau masalahnya bukan tentang transaksi spesifik.
  • Deskripsi Masalah (wajib) — text area bebas, minimal 20 karakter.
Yang Terjadi di Sistem (Validasi Real-time)
Saat user mengetik, app langsung cek format nomor WA (harus 08xx atau +628xx, 10–13 digit). Tombol "Kirim Laporan" tetap abu-abu (tidak bisa diklik) sampai semua field wajib terisi dengan benar.
3
User tap "Kirim Laporan"
Setelah semua field terisi, user tap tombol kirim. Sistem langsung proses dan buat tiket.
👤 User ⚙️ Sistem (BE)
Yang User Lihat
Tombol berubah jadi loading spinner sebentar (1–2 detik). Tidak bisa di-tap dua kali — ini mencegah tiket duplikat.
Yang Terjadi di Sistem
Sistem terima data form → buat tiket baru di database dengan info:
  • Nomor tiket unik, contoh: #TKT-20240628-001
  • Status awal: Menunggu
  • Priority: ditentukan otomatis berdasarkan kategori (Transaksi Gagal = High, Lainnya = Normal)
  • Assign otomatis ke CS yang sedang bertugas
Setelah tiket tersimpan, sistem langsung kirim WA ke nomor yang user isi.
4
User terima konfirmasi
Dua konfirmasi langsung dikirim: di dalam app/web dan via WhatsApp.
👤 User ⚙️ Sistem
Di App/Web (Screen "Tiket Terkirim")
Screen berganti ke halaman konfirmasi yang menampilkan:
  • ✅ Ikon sukses besar
  • Nomor tiket: #TKT-20240628-001
  • Pesan: "Laporan kamu sudah kami terima. Tim kami akan menghubungi kamu via WhatsApp dalam 1×24 jam."
  • Tombol "Kembali ke Beranda"
Via WhatsApp (Otomatis)
Pesan WA dikirim ke nomor yang diisi:

"Halo! Laporan kamu sudah kami terima 🙏
Nomor tiket: #TKT-20240628-001
Kategori: Transaksi Gagal

Tim GoPop akan menghubungi kamu di nomor ini dalam 1×24 jam. Simpan nomor tiket ini ya!"
Bottleneck & Cara Mengatasinya
⚡
3 Case yang Bisa Bermasalah di Flow Ini
User isi nomor WA yang salah atau tidak aktif — CS tidak bisa menghubungi
Tambahkan langkah verifikasi ringan: setelah user isi nomor WA, kirim kode 4 digit ke nomor itu. User input kode untuk konfirmasi nomor aktif. Kalau nomor salah, user tahu sebelum tiket disubmit. Kode ini berbeda dengan OTP login — tidak mempengaruhi akun user.
👤 Keputusan: Bili (apakah verifikasi nomor WA perlu?) · ⚙️ Implementasi: Irwan + Rifky (BE) + Agung/Fikly (FE)
WA konfirmasi tidak terkirim karena API WA belum ada
Ini gap yang sudah teridentifikasi (INT-GAP-10). Sementara WA API belum ready, konfirmasi hanya lewat screen app. User tetap dapat nomor tiket — jangan tampilkan "WA sudah dikirim" sebelum benar-benar dikirim. Tampilkan saja: "Tiket #TKT-xxx berhasil dibuat."
⚙️ Fix: Irwan + Rifky (implementasi WA API dulu, baru aktifkan notif WA untuk tiket)
User double-tap tombol "Kirim" dan tiket terduplikat
Sistem harus tolak request identik yang masuk dalam 30 detik (same user/nomor WA + same kategori). Di FE, tombol langsung di-disable setelah pertama kali di-tap. Di BE, ada pengecekan duplikat sebelum buat tiket baru.
⚙️ Implementasi: Agung/Fikly (disable button di FE) + Irwan + Rifky (idempotency check di BE)
✅ Hasil Akhir Flow Ini
Tiket tersimpan di sistem dengan nomor unik dan status "Menunggu"
CS on-duty otomatis di-assign dan bisa lihat tiket di dashboard
User dapat nomor tiket sebagai bukti laporan mereka diterima
WA konfirmasi dikirim ke nomor yang user isi (setelah INT-GAP-10 selesai)

Komplain & Tiket › D-F02
📥 CS Terima & Review Tiket Baru
Setelah user submit tiket, tiket masuk ke dashboard CS. Flow ini menjelaskan bagaimana CS tahu ada tiket baru, bagaimana membuka dan membacanya, serta apa yang CS lihat di dalamnya.
🎧 CS GoPop 🔔 Trigger: Tiket baru masuk dari user ⏱ Target Respons: <1 jam (High Priority)
Alur Step by Step
1
CS dapat notifikasi tiket baru
Tiket yang baru masuk langsung muncul di dashboard CS tanpa perlu refresh halaman.
🎧 CS ⚙️ Sistem
Di Dashboard CS
Badge merah di menu "Komplain & Tiket" bertambah — misalnya dari "3" jadi "4". Kalau CS sedang buka halaman lain, ada notifikasi pop-up kecil di pojok kanan bawah: "📥 Tiket baru dari +628xx — Transaksi Gagal".
Penentuan Priority Otomatis
Sistem tentukan priority tiket secara otomatis berdasarkan kategori:
  • Urgent — Transaksi Gagal dengan nominal > Rp 500.000
  • High — Transaksi Gagal, Saldo Tidak Masuk
  • Normal — Masalah Akun/Login, Lainnya
Tiket Urgent dan High muncul di bagian atas list, dengan label warna merah atau oranye.
2
CS buka list tiket dan pilih yang akan dihandle
CS masuk ke halaman Komplain & Tiket. Tiket ditampilkan berurutan dari yang paling urgent.
🎧 CS
Yang CS Lihat di List Tiket
Setiap baris tiket menampilkan:
  • Nomor tiket + waktu masuk ("5 menit yang lalu")
  • Kategori masalah + badge priority (merah/oranye/abu)
  • Nomor WA user (disamarkan: 0812-xxxx-8901)
  • Cuplikan deskripsi masalah (max 60 karakter)
  • Status saat ini (Menunggu / Diproses / Selesai)
CS bisa filter: "Semua" / "Belum Dihandle" / "Milik Saya" / "Urgent".
3
CS klik tiket untuk lihat detail
CS buka tiket dan baca semua informasi yang user kirimkan, termasuk konteks transaksi jika ada.
🎧 CS ⚙️ Sistem
Yang CS Lihat di Detail Tiket
Panel detail tiket terbuka di sebelah kanan (atau halaman baru di mobile):
  • Semua info form yang user isi (nomor WA, kategori, deskripsi)
  • Kalau user mengisi ID Transaksi → otomatis muncul ringkasan transaksi itu: produk, nominal, status, waktu transaksi
  • Timeline aktivitas tiket (kapan dibuat, siapa yang di-assign)
  • Field "Catatan Internal" — hanya CS/admin yang bisa lihat, user tidak tahu
Yang Terjadi di Sistem
Saat CS buka tiket, status otomatis berubah dari Menunggu ke Dibaca. Waktu "dibaca pertama kali" dicatat — berguna untuk laporan SLA nanti.
4
CS mulai proses tiket
CS tap "Mulai Proses" untuk menandai tiket sedang ditangani. Status berubah, tiket terkunci ke CS ini.
🎧 CS
Aksi CS
CS tap tombol "Mulai Proses" di pojok kanan atas detail tiket. Status berubah ke Sedang Diproses. Nama CS otomatis tercatat sebagai PIC tiket ini. CS lain bisa lihat tapi tidak bisa ambil alih kecuali CS ini melepasnya.
Bottleneck & Cara Mengatasinya
⚡
3 Case yang Bisa Bermasalah di Flow Ini
CS sedang offline atau sibuk — tiket menunggu terlalu lama
Sistem perlu pantau tiket yang belum dibuka dalam 30 menit. Kalau lewat 30 menit dan CS yang di-assign tidak membuka tiket, sistem auto-reassign ke CS lain yang sedang aktif (online di dashboard). Kalau semua CS offline, kirim notifikasi ke Acoy sebagai monitor.
⚙️ Implementasi: Irwan + Rifky (cron job pantau tiket) + Isal/Ipong (dashboard notif ke Acoy)
Tiket masuk tapi tidak ada ID Transaksi — CS tidak tahu kasus mana
CS bisa cari manual dari dashboard: tombol "Cari Transaksi" di dalam detail tiket. CS input nomor HP user atau rentang waktu kejadian → sistem tampilkan kandidat transaksi yang mungkin terkait. CS pilih yang paling sesuai dan "link" ke tiket ini.
⚙️ Implementasi: Isal/Ipong (fitur cari dan link transaksi di detail tiket) + Irwan/Rifky (endpoint search)
Dua CS buka tiket yang sama bersamaan — bisa terjadi duplikat penanganan
Saat CS pertama tap "Mulai Proses", tiket langsung terkunci untuk CS itu. CS kedua yang buka tiket yang sama akan lihat banner: "Tiket ini sedang ditangani oleh [nama CS]. Kamu bisa baca tapi tidak bisa mengambil alih." Hanya Acoy (Super CS) yang bisa force-reassign.
⚙️ Implementasi: Irwan/Rifky (locking logic di BE) + Isal/Ipong (UI banner lock)
✅ Hasil Akhir Flow Ini
CS sudah membaca dan memahami masalah user
Tiket berstatus "Sedang Diproses" dan terkunci ke CS yang handle
Kalau ada ID Transaksi, CS sudah bisa lihat detail transaksi terkait
Waktu respons pertama tercatat untuk laporan SLA

Komplain & Tiket › D-F03
🔍 CS Investigate & Selesaikan Tiket
CS sudah buka tiket dan mulai proses. Sekarang CS perlu cari tahu apa masalahnya, hubungi user via WA untuk update, dan selesaikan tiket dengan resolusi yang tepat.
🎧 CS GoPop 🔔 Trigger: Tiket status "Sedang Diproses" ⏱ Target Selesai: <24 jam
Alur Step by Step
1
CS cek status transaksi terkait
Langkah pertama CS selalu cek dulu di sistem — apakah transaksi memang bermasalah, atau sudah selesai tapi user belum tahu.
🎧 CS
Yang CS Lakukan di Dashboard
CS klik "Lihat Detail Transaksi" di dalam detail tiket → langsung dibawa ke halaman detail transaksi di modul Monitoring. CS lihat: status saat ini, waktu terakhir diupdate, sudah berapa kali retry, apakah sudah ada refund yang berjalan.
2
CS hubungi user via WA dengan update status
CS tidak balas via dashboard — CS hubungi user langsung ke nomor WA yang mereka isi di form. CS bisa pakai template pesan yang sudah disediakan.
🎧 CS 👤 User
Yang CS Lakukan
Tap tombol "Hubungi via WA" di detail tiket → nomor WA user langsung terbuka di WA CS. CS pilih template pesan sesuai kondisi, lalu kirim. Setelah menghubungi, CS tap "Tandai Sudah Dihubungi" di dashboard — waktu kontak dicatat.
Template Pesan yang Tersedia
  • "Halo [nama]! Ini CS GoPop terkait tiket #TKT-xxx. Transaksi kamu sedang kami proses, dana kamu aman. Estimasi selesai [waktu]."
  • "Halo [nama]! Kami sudah cek tiket #TKT-xxx. Transaksi berhasil — [nama produk] sudah terkirim ke [tujuan]. Ada yang bisa kami bantu lagi?"
  • "Halo [nama]! Terkait tiket #TKT-xxx, kami perlu escalate ke tim terkait. Estimasi respons 2×24 jam. Kami akan update kamu via WA ini."
3
Masalah selesai — CS tutup tiket
Kalau masalah sudah terselesaikan (transaksi sukses, refund sudah berjalan, atau user puas dengan penjelasan), CS tutup tiket dengan memilih jenis resolusi.
🎧 CS ⚙️ Sistem
Yang CS Lakukan
CS tap "Tutup Tiket" → muncul form penutupan yang wajib diisi:
  • Jenis Resolusi (dropdown wajib): Transaksi Berhasil / Refund Diproses / Edukasi User / Tidak Ada Masalah / Lainnya
  • Catatan Penutupan (teks opsional): ringkasan apa yang terjadi dan bagaimana diselesaikan
Setelah submit → tiket status berubah ke Selesai. Tidak bisa diubah lagi kecuali user re-open (flow D-F05).
Yang Terjadi di Sistem
Waktu selesai dicatat. Total durasi penanganan dihitung otomatis (dari tiket dibuat sampai ditutup). Data ini masuk ke laporan SLA bulanan Acoy.
Bottleneck & Cara Mengatasinya
⚡
3 Case yang Bisa Bermasalah di Flow Ini
CS tidak punya akses untuk lihat detail transaksi — harus minta ke tim lain
CS harus bisa lihat detail transaksi (status, waktu, produk, nominal) tapi tidak boleh lihat data sensitif seperti HPP reseller atau margin. Di dashboard, CS punya akses read-only ke Monitoring Transaksi. Tidak perlu minta ke Rifky atau Acoy hanya untuk cek status.
⚙️ Implementasi: Irwan/Rifky (RBAC — CS role bisa GET /transactions tapi tidak bisa aksi) + Isal/Ipong (shortcut link dari tiket ke transaksi)
User tidak balas WA — CS tidak tahu apakah masalah sudah selesai
CS tetap bisa tutup tiket meski user tidak balas, dengan resolusi "Tidak Ada Respons dari User setelah 3×24 jam". Sebelum tutup, CS kirim WA terakhir: "Tiket #TKT-xxx akan kami tutup dalam 24 jam. Balas WA ini kalau masalah belum selesai." Kalau user balas setelah ditutup, bisa re-open (flow D-F05).
👤 Keputusan: Bili (berapa lama tunggu sebelum auto-close?) · ⚙️ Isal/Ipong (reminder auto-close di dashboard)
CS tidak tahu harus jawab apa untuk kasus teknikal tertentu
Tambahkan knowledge base sederhana di dashboard: CS bisa cari berdasarkan kategori ("token listrik gagal", "transfer ewallet pending") dan dapat panduan respon standar. Ini juga memastikan jawaban CS konsisten. Knowledge base bisa dikelola Acoy atau Bili dari CMS.
📋 Keputusan: Acoy (buat knowledge base awal) · ⚙️ Isal/Ipong (fitur search KB di detail tiket) · 👤 Bili (integrasi ke CMS)
✅ Hasil Akhir Flow Ini
Tiket berstatus Selesai dengan jenis resolusi yang tercatat
User sudah dihubungi via WA dan tahu statusnya
Durasi penanganan tercatat untuk laporan SLA
Kalau belum selesai sendiri → lanjut ke F04 (Eskalasi)

Komplain & Tiket › D-F04
📤 Eskalasi ke Ops atau Finance
Ada kasus yang CS tidak bisa selesaikan sendiri — seperti refund yang gagal, transaksi yang perlu force update status, atau kasus yang butuh akses lebih tinggi. Flow ini menjelaskan bagaimana eskalasi dilakukan dan apa yang terjadi setelahnya.
🎧 CS → Ops (Acoy) / Finance (Agung) 🔔 Trigger: CS tidak bisa resolve sendiri ⏱ Target Respons Eskalasi: <2 jam
Kapan Harus Eskalasi?
📋
Eskalasi ke Ops (Acoy) — kalau butuh: cek atau retry transaksi ke biller, force update status transaksi, investigasi transaksi stuck >1 jam.

Eskalasi ke Finance (Agung) — kalau butuh: proses refund manual, approve refund di atas batas nominal CS, investigasi selisih saldo reseller.
Alur Step by Step
1
CS memutuskan untuk eskalasi
Setelah CS cek dan tidak bisa selesaikan, CS tap tombol "Eskalasi" di detail tiket.
🎧 CS
Yang CS Lakukan
Tap "Eskalasi Tiket" → muncul form eskalasi:
  • Eskalasi ke (wajib pilih): Ops (Acoy) / Finance (Agung) / IT (Irwan)
  • Alasan Eskalasi (wajib isi): CS jelaskan apa yang sudah dicoba dan kenapa butuh bantuan
  • Prioritas: Normal / Urgent (kalau user sudah komplain lebih dari 1 kali atau nominal besar)
2
Ops/Finance dapat notifikasi eskalasi
Orang yang di-eskalasi langsung dapat notif di dashboard dan via WA internal.
👁️ Acoy (Ops) 💰 Agung (Finance) ⚙️ Sistem
Yang Terjadi di Dashboard
Tiket muncul di halaman "Tiket yang Di-eskalasi ke Saya" dengan label ESCALATED. Badge di menu Komplain bertambah untuk Acoy atau Agung. Mereka bisa lihat: deskripsi masalah user, riwayat yang sudah dilakukan CS, dan alasan eskalasi.
WA Notif Internal
"[Nama CS] mengalihkan tiket #TKT-xxx ke kamu.
Kategori: Transaksi Gagal (Refund Manual)
Alasan: Auto-refund gagal, butuh proses manual.
Prioritas: Urgent
Buka dashboard untuk tindak lanjut."
3
Ops/Finance tindak lanjut dari dashboard
Acoy atau Agung buka tiket, cek detail transaksi, lalu lakukan aksi yang diperlukan — langsung dari dashboard tanpa perlu koordinasi manual.
👁️ Acoy (Ops) 💰 Agung (Finance)
Aksi yang Bisa Dilakukan
Ops (Acoy):
  • Retry transaksi ke biller
  • Cek status ke biller (pull status manual)
  • Update status transaksi dengan alasan
Finance (Agung):
  • Proses refund manual ke metode bayar asal
  • Input nominal + konfirmasi → sistem proses refund via Xendit
  • Tambahkan catatan untuk laporan internal
4
Update dikirim ke CS dan user
Setelah Ops/Finance selesai handle, CS yang awal menangani dapat notif update. CS yang follow up ke user.
🎧 CS 👤 User ⚙️ Sistem
Alur Balik ke CS
Ops/Finance klik "Selesai dari Sisi Saya" + isi catatan apa yang dilakukan → tiket kembali ke CS dengan status "Perlu Follow-up ke User". CS dapat notif: "Eskalasi #TKT-xxx sudah ditangani Agung. Refund sudah diproses. Hubungi user untuk konfirmasi."
WA ke User (Dikirim CS)
"Halo [nama]! Update terkait tiket #TKT-xxx: refund sebesar Rp [nominal] sudah kami proses. Dana akan kembali ke [metode bayar] dalam 1–3 hari kerja. Ada pertanyaan? Balas WA ini ya!"
Bottleneck & Cara Mengatasinya
⚡
2 Case yang Bisa Bermasalah di Flow Ini
Acoy atau Agung tidak buka dashboard — eskalasi tidak ditindaklanjuti
Kalau eskalasi Urgent tidak ada respons dalam 1 jam, sistem kirim pengingat WA kedua ke yang bersangkutan. Kalau 2 jam masih tidak ada respons, notif juga ke Acoy (sebagai monitor semua). Semua eskalasi terlambat masuk ke laporan mingguan yang Acoy review.
⚙️ Implementasi: Irwan/Rifky (cron job pantau eskalasi) + Isal/Ipong (laporan eskalasi di dashboard)
Eskalasi bolak-balik antar tim — tiket ping-pong dan user tidak dapat kepastian
Setiap eskalasi harus ada alasan yang jelas dan aksi konkret. Kalau Agung terima eskalasi tapi butuh info dari Acoy, mereka communicate via catatan internal di tiket (bukan eskalasi lagi). Hanya boleh eskalasi 1x per level — CS → Ops/Finance. Tidak ada CS → Ops → Finance → Ops lagi.
📋 Keputusan: Acoy (buat SOP eskalasi yang jelas) · ⚙️ Isal/Ipong (batasi eskalasi max 1x di sistem)
✅ Hasil Akhir Flow Ini
Masalah yang butuh akses lebih tinggi sudah ditangani Ops atau Finance
Semua aksi tercatat di timeline tiket — ada audit trail lengkap
User sudah di-update via WA dengan status terbaru
CS tutup tiket setelah konfirmasi user puas → selesai

Komplain & Tiket › D-F05
🔄 Tiket Re-open Setelah Ditutup
Kadang tiket sudah ditutup tapi masalah ternyata belum benar-benar selesai. User bisa re-open tiket yang sama tanpa perlu buat tiket baru — semua riwayat percakapan sebelumnya tetap ada.
👤 User + 🎧 CS 🔔 Trigger: Masalah belum selesai setelah tiket ditutup ⏱ Window Re-open: 7 hari setelah ditutup
Alur Step by Step
1
User minta buka kembali tiket
User bisa re-open lewat dua cara: balas WA dari CS, atau buka Riwayat Tiket di app dan tap "Buka Kembali".
👤 User
Cara 1 — Balas WA CS
User balas WA terakhir dari CS dengan pesan apapun dalam 7 hari setelah tiket ditutup → sistem deteksi balasan ini → tiket otomatis re-open dan CS yang sama mendapat notif.
Cara 2 — Dari App (Riwayat Tiket)
User buka menu Riwayat Tiket → pilih tiket yang statusnya "Selesai" → tap tombol "Masalah Belum Selesai? Buka Kembali" → isi penjelasan singkat kenapa re-open.
2
Tiket re-open dengan konteks lengkap
Tiket yang sama dibuka kembali — bukan buat tiket baru. Semua riwayat CS, eskalasi, dan aksi sebelumnya masih ada.
⚙️ Sistem 🎧 CS
Yang CS Lihat Saat Buka Tiket Re-open
Tiket muncul kembali di list dengan label RE-OPENED warna oranye. CS yang sama otomatis di-assign ulang. Di dalam tiket, ada section baru "Alasan Re-open" yang menampilkan penjelasan user. Semua riwayat lama tetap tersedia di bawahnya — CS tidak perlu tanya ulang dari awal.
Aturan Re-open di Sistem
  • Hanya bisa re-open maksimal 3 hari setelah tiket ditutup
  • Kalau sudah lebih dari 3 hari, user harus buat tiket baru (tapi bisa referensikan nomor tiket lama)
  • Satu tiket bisa di-reopen maksimal 2 kali. Lebih dari itu → otomatis di-escalate ke Acoy
3
CS handle ulang dan selesaikan
CS lihat alasan re-open, cek ulang apa yang sudah dilakukan sebelumnya, dan tindaklanjut dengan info baru dari user.
🎧 CS 👤 User
Yang CS Lakukan
CS baca alasan re-open → cek kembali status transaksi yang relevan → hubungi user via WA dengan update terbaru. Kalau masalahnya berbeda dari sebelumnya → bisa eskalasi lagi (flow D-F04). Kalau sudah selesai → tutup tiket lagi dengan resolusi baru ditambahkan.

2 Case yang Bisa Bermasalah di Flow Ini
User yang sama re-open tiket berkali-kali padahal masalah sudah selesai
Setelah re-open ke-2, sistem otomatis flag tiket dan Acoy di-cc. Acoy bisa lihat pola — apakah ini user yang genuinely punya masalah, atau ada miscommunication. Kalau pola mencurigakan (mis. user coba dapat refund dua kali untuk transaksi yang sama), Acoy bisa mark tiket sebagai "Closed - No Further Action" dan blokir re-open.
👁️ Acoy (review tiket bermasalah) · ⚙️ Isal/Ipong (flag re-open ke-2 di sistem)
CS berbeda yang handle re-open — tidak tahu konteks sebelumnya
Sistem prioritaskan assign re-open ke CS yang sama dengan yang handle sebelumnya. Kalau CS itu sedang tidak aktif, sistem assign ke CS lain tapi dengan catatan di atas tiket: "Tiket ini sebelumnya ditangani oleh [nama CS]. Baca riwayat di bawah sebelum hubungi user." Riwayat lengkap selalu tersedia.
⚙️ Implementasi: Irwan/Rifky (preferential assign di BE) + Isal/Ipong (banner konteks di UI)
